  1. UB-S(outcome) 和 “纯结构上界” 不能完全画等号
     这个我认同。
     因为如果只用 branchPC + exact outcome history 做 key，它已经顺手把一部分 path alias 也消掉了，所以它更像：

  - 一个很强的、偏乐观的 structural upper bound
  - 而不是“只去掉 set/way 冲突，其它都不变”的最小上界

  所以命名上最好更明确。
  我倾向于把第一版直接叫：

  - UB-S(outcome) 或
  - Per-Branch Exact-Outcome-History TAGE

  这样后面如果再做 UB-SP-lite(exact path)，语义会更清楚。

  2. 独立新类最好继承 BTBTAGE
     这个我也认同，而且这是个很关键的工程判断。
     review 说得对，当前 frontend 装配里本来就是围绕 BTBTAGE* 和 Param.BTBTAGE(...) 走的。
     如果完全独立不继承，改动面会不必要地放大。

  所以第一版最合理的形态应该是：

  - BTBTAGEUpperBound : public BTBTAGE

  复用这些现成能力：

  - history/spec update/recover 生命周期
  - TagePrediction
  - TageMeta
  - predictor integration path

  只替换：

  - provider/alt lookup
  - exact-key storage
  - allocation path

  这会比“全新平行实现一个 predictor”稳很多。

  3. updateOnRead 第一版应该关掉
     这个我也认同。
     对 current array-based TAGE，updateOnRead 的重查代价还行；
     但对 exact-history map 版，它会把最重的查找路径付两次。

  如果 upper-bound 的目标是“判别空间”，不是“完全复制当前微观行为”，那第一版更实用的做法是：

  - prediction 阶段把 provider/alt/key/table 存进 meta
  - update 阶段直接用 meta
  - v1 关闭 updateOnRead

  这是非常合理的 tradeoff。

  4. 内存和运行时间风险比我原文写得更重
     这个我也认同。
     尤其是后几张长历史表，如果 key 真做成 exact-history，unordered_map 的 node/bucket/pointer 开销会比 entry 本身还显眼。

  所以如果做实现，我会把下面这些当成 v1 必做项，而不是“优化项”：

  - 固定宽度 key，不用 dynamic_bitset 当 map key
  - 自定义 hash
  - 每表 reserve
  - 统计 contexts_per_table
  - 统计 contexts_per_hot_branch

  这一点 review 说得很对：
  这个 upper bound 最大的风险不是“写不出来”，而是“写出来后太慢/太大，结果还不好解释”。

  我只有一个地方会稍微补充，而不是反对：

  第一版不一定非得马上补完整单测体系，但至少要补 very small sanity tests。
  这点和 review 本质一致，只是我会把范围收得更小：

  - exact key lookup 对不对
  - provider/alt 选择对不对
  - recover 后 key 能不能对齐 prediction-time meta

  不用一开始就追求把现有 BTBTAGE 单测体系平移过去。
