**Findings**

1. 高危: speculative history 仍然是全局的，且只按 `streamId/ftqId` 索引，两个线程会互相提交/回滚对方的历史。
[decoupled_bpred.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.hh#L143) 只保留了单个 `HistoryManager historyManager`。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L917) 往 history manager 写入时只传 `ftq.backId(tid) + 1`，没有带 `tid`。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L547) 和 [decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L966) 在 commit/squash 时同样只按 `target_id` 操作。
[history_manager.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/history_manager.hh#L75) 的内部表也是单个 `speculativeHists`，而 [history_manager.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/history_manager.hh#L127) / [history_manager.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/history_manager.hh#L170) 的匹配条件都只有 `streamId`。
FTQ 现在是 per-thread，两个线程都会从 `1` 开始编号，所以 `tid0` 提交 `ftqId=1` 时，会把 `tid1` 的 `ftqId=1` 也一起提交/回滚。后果是 GHR/RAS 恢复错乱，严重时会直接触发 `checkHistory()` 断言。

2. 高危: BPU 侧根本没有真正做 SMT 调度，预测永远只给 `tid 0` 生成。
[decoupled_bpred.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.hh#L147) 的 `scheduleThread()` 直接 `return 0`。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L124) 在 `tick()` 中只对 `curTid` 发起 `requestNewPrediction()`，因此只有 `tid 0` 会变成 `validprediction=true`。
[fetch.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L1749) 又改成从 FTQ 里选可取指线程；如果 `tid 1` 永远没有 FTQ entry，它就永远拿不到前端带宽。
这不是“调度策略还没调优”，而是功能上 thread 1 会直接饿死。

3. 高危: `fetch.cc` 仍然在多个关键路径偷用 `tid 0` 的 FTQ 头，线程 1 会读错流，甚至直接断言。
[fetch.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L621) 在 cache completion 对齐检查里写的是 `dbpbtb->ftqHasFetching(0)` / `ftqFetchingTarget(0)`，但实际比较的是 `threads[tid].startPC`。
[fetch.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L758) 在 `lookupAndUpdateNextPC()` 里先 `assert(dbpbtb->ftqHasFetching(0))`，然后却读取 `ftqFetchingTarget(tid)`。
如果 thread 1 有自己的 entry 而 thread 0 没有，这里会误报断言；如果两个线程都有 entry，这里会把 thread 1 的 fetch buffer 跟 thread 0 的 stream 做一致性检查，直接造成错误 panic 或错误 next-PC。

4. 中高危: resolve-backpressure 仍然是全局状态，而且只会错误地阻塞 `tid 0`。
[decoupled_bpred.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.hh#L144) 的 `resolveDequeueFailCounter` 仍是单个全局计数。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L623) 的 `notifyResolveFailure()` 没有 `tid` 参数。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L634) 的 `blockPredictionOnce()` 还留着 `// smtTODO`，并且硬编码 `threads[0].blockPredictionPending = true`。
结果是 thread 1 的 resolved-update 堵塞会去停 thread 0 的预测，而真正出问题的 thread 1 反而不一定被限流。这会制造跨线程性能串扰，且很难从 stats 里看出来。

5. 中危: prediction trace/DB 记录的 FTQ ID 是错的，首条会记成 0，后续整体滞后一条。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L379) 在 `ftq.insert(entry)` 之前就调用 `PredictionTrace(ftq.backId(tid), entry)`。
[ftq.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/ftq.hh#L46) 的 `backId()` 定义为 `baseTargetId + size - 1`。
队列为空时第一条 prediction 的 `backId()` 是 `0`；队列非空时拿到的是“上一条”的 ID，不是当前新 entry 的 ID。所有基于这个 trace 做 FSQ/FTQ 对账、误预测归因、数据库关联的脚本都会被误导。

6. 中危: 线程数和 reset 语义仍然是硬编码的，配置和行为都不自洽。
[decoupled_bpred.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.hh#L78) 把 `numThreads` 写死成 `2`。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L48) 也把 `FetchTargetQueue` 构造成 `ftq(2, p.ftq_size)`。
[decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/decoupled_bpred.cc#L865) 的 `resetPC()` 会把同一个 `new_pc` 灌给所有线程。
[fetch.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L453) reset 时传入的还是 `threads[0].fetchpc`。
这意味着即使后面把 `scheduleThread()` 修了，thread 1 在 reset/trace init 后仍会从 thread 0 的 PC 开始建预测状态；另外实现也无法正确支持 `numThreads != 2` 的配置。
