**1. 先补 GEM5 O3 的基本图景**

gem5 的 O3 主流水线，你先可以把它粗略理解成：

```text
指令前行方向:
Fetch -> Decode -> Rename -> IEW -> Commit

其中 IEW = Issue / Execute / Writeback 的组合区
Commit 负责按程序序退休，ROB 也主要在这边管理
```

每一级做的事大致是：

- `Fetch`：取指、和分支预测器/FTQ 打交道。
- `Decode`：解码，做早期分支校验、一些融合。
- `Rename`：把架构寄存器映射到物理寄存器，处理重命名资源。
- `IEW`：接收 rename 后的指令，分发到 IQ/LSQ/FU，执行并回写。
- `Commit`：按顺序从 ROB 退休，处理 trap、squash、精确异常。

关键代码入口在 [src/cpu/o3/cpu.cc:567](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/cpu.cc#L567)。

**2. `timeBuffer` 到底是什么**

这是理解这个 PR 最关键的概念。

`TimeBuffer` 的实现本质上是一个“带过去/未来槽位的环形缓冲”，你可以把它看成“可配置延迟的流水寄存器链”。

关键位置在 [src/cpu/timebuf.hh:40](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/timebuf.hh#L40)、[src/cpu/timebuf.hh:179](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/timebuf.hh#L179)、[src/cpu/timebuf.hh:232](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/timebuf.hh#L232)。

核心语义是：

- `getWire(0)`：访问“当前周期写入槽”
- `getWire(-delay)`：访问“延迟 `delay` 个周期后才能看到的旧槽”
- `advance()`：周期结束时整体推进一格

所以它不是普通 C++ 变量，不是“你写了别人立刻就看见”。

它更像：

```text
cycle N:
  producer 往 slot(0) 写
  consumer 读 slot(-delay)

cycle 末尾:
  advance()

cycle N+1:
  上一拍 producer 写的内容，才成为 consumer 看到的 slot(-1)
```

这就是为什么 gem5 里大量代码是：

- 后级读 `fromX = timeBuffer->getWire(-someDelay)`
- 前级写 `toY = timeBuffer->getWire(0)`

例如：

- Fetch 从 Decode/ Rename / IEW / Commit 收回退控制信息  
  [src/cpu/o3/fetch.cc:368](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L368)
- Decode 写回 Fetch，读 Fetch->Decode 队列  
  [src/cpu/o3/decode.cc:177](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/decode.cc#L177)
- Rename 读 Commit/IEW 的反馈，写回 Decode  
  [src/cpu/o3/rename.cc:208](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/rename.cc#L208)

**4. 旧模型为什么会越来越难维护**

从 diff 看，旧代码里 Decode/Rename 这类 stage 有很重的状态机：

- `Blocked`
- `Unblocking`
- `Squashing`
- `SerializeStall`
- `skidBuffer`
- 以及 block/unblock 握手位

这类状态在旧版 `decode.hh` / `rename.hh` 里非常明显。

它的问题不是“不能工作”，而是：

1. 上游先 tick，下游后 tick  
   所以上游当拍做决定时，天然不知道下游“这拍刚刚”变成拥塞还是恢复。
2. 为了补这个时序差，只能引入一堆“我已经 block 了”“我准备 unblock 了”“我正在把旧指令塞回 skid buffer”这样的中间状态。
3. 一旦做 SMT，这些状态要按线程复制，而且不同 stage 还要彼此对齐，复杂度会爆炸。

也就是说，旧模型更像是：

**“用软件状态机去补模拟器 tick 顺序带来的时序不直观”**

而这次 PR 想做的是：

**“把 tick 顺序和阻塞传播方式改得更接近真正的后压流水线”**

**5. 这次 PR 的核心改变是什么**

我认为有 4 个核心点。

**5.1 `CPU::tick()` 改成反向顺序**

以前大致是：

```text
fetch -> decode -> rename -> iew -> commit
```

现在变成：

```text
commit -> iew -> rename -> decode -> fetch
```

代码在 [src/cpu/o3/cpu.cc:582](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/cpu.cc#L582)。

这件事的含义不是“指令方向反了”。  
指令还是从 Fetch 往 Commit 走。  
变的是**软件里每拍谁先更新自己的状态**。

直觉上就是：

- 先让后级看自己这拍能不能收、有没有 squash、ROB 满不满
- 再让前级根据这个决定自己这拍送不送

这非常符合“后压”直觉。

5. `fetch.tick()` 最后跑

6. 周期末统一 `advance()`

## Core Idea

This PR is mainly about changing how the O3 pipeline models timing and backpressure.

In one sentence:

- instructions still move forward as `Fetch -> Decode -> Rename -> IEW -> Commit`
- backpressure now propagates in the opposite direction as `Commit -> IEW -> Rename -> Decode -> Fetch`

The main purpose is to make stage blocking more direct and more suitable for later SMT-oriented per-thread flow control.

## Pipeline Background

The O3 frontend/backend path is roughly:

```text
Fetch -> Decode -> Rename -> IEW -> Commit
```

Where:

- `Fetch`: fetch instructions and interact with branch prediction / FTQ
- `Decode`: decode instructions, do early control-flow checks and fusion
- `Rename`: allocate physical registers and rename architectural state
- `IEW`: dispatch / issue / execute / write back
- `Commit`: retire instructions in-order through ROB

Key implementation:

- `src/cpu/timebuf.hh`

- `getWire(0)`: write/read the current slot
- `getWire(-delay)`: read a value that becomes visible only after the configured delay
- `advance()`: move the time window at cycle boundary

- `fetchTimebuffer`
- `decodeTimebuffer`

- squash / redirect info

## Old Tick Order vs New Tick Order

Before this PR, the CPU tick order was roughly:

```text

```

After this PR, the order becomes:

```text
commit -> iew -> rename -> decode -> fetch
```

- `src/cpu/o3/cpu.cc`

This does **not** mean instructions move backward.
It means the simulator now lets later stages decide first whether they can accept work, then earlier stages react in the same cycle.

Under the old order:

- frontend stages ran first
- backend stages discovered congestion later in the same cycle
- therefore the frontend only learned about some blocking conditions one cycle later

To compensate, the old code had a larger amount of explicit stage state machinery:

- blocked / unblocking / squashing / serialize-stall style states
- skid buffers
- block/unblock handshake signals in the delayed control path

That approach can work, but it becomes difficult to reason about and difficult to extend to SMT.

With the new reverse tick order:

1. `Commit` decides whether IEW must stop
2. `IEW` immediately decides whether Rename must stop
3. `Rename` immediately decides whether Decode must stop
4. `Decode` immediately decides whether Fetch must stop
5. `Fetch` sees the final backpressure in the same cycle

This is the conceptual backpressure chain:

```text
Commit -> IEW -> Rename -> Decode -> Fetch
```

This PR does not fully implement SMT scheduling, but it changes the structure so that SMT becomes easier to support.

Reasons:

1. `StallSignals` are already per-thread arrays
2. multiple stages now use per-thread local buffers before consuming input
3. the backpressure chain is explicit, rather than spread across many state transitions

So the PR is best understood as:

- not “SMT completed”
- but “pipeline control reorganized so SMT-friendly blocking becomes feasible later”

# PR 背景说明：Reverse Ordered Tick 与 Stall 重构

## 核心结论

这个 PR 的主线不是改某个具体算法，而是重构 O3 流水线的时序建模方式。

一句话概括：

- 指令仍然沿着 `Fetch -> Decode -> Rename -> IEW -> Commit` 向前流动
- 阻塞/反压则改成沿着 `Commit -> IEW -> Rename -> Decode -> Fetch` 反向传播

主要目标是让各级阻塞关系更直接、更接近真实流水线，也为后续 SMT 下按线程、按级传播背压做结构准备。

## O3 流水线基本背景

这个 O3 CPU 的主链路可以先粗略理解为：

```text
Fetch -> Decode -> Rename -> IEW -> Commit
```

各级职责大致如下：

- `Fetch`：取指，和分支预测器 / FTQ 交互
- `Decode`：解码，做部分早期控制流检查和融合
- `Rename`：进行物理寄存器分配与重命名
- `IEW`：分发、发射、执行、写回
- `Commit`：通过 ROB 按程序序退休

总入口在：

- `src/cpu/o3/cpu.cc`

## `TimeBuffer` 是什么

`TimeBuffer` 不是普通共享变量，它更像“显式带延迟的流水寄存器链”。

关键实现：

- `src/cpu/timebuf.hh`

你可以这样理解：

- `getWire(0)`：访问当前时刻的槽位
- `getWire(-delay)`：读取延迟 `delay` 拍后才能看到的数据
- `advance()`：在周期边界推进整个窗口

所以它的作用不是简单“共享数据”，而是用来精确建模 stage 之间的信息传播延迟。

- `decodeTimebuffer`
- `renameTimebuffer`
- `iewTimebuffer`

### 2. 反向控制/反馈流

- `src/cpu/o3/comm.hh`

- squash / redirect 信息

- `src/cpu/o3/comm.hh`

这是这次重构最关键的点。

新的职责划分变成：

- `TimeBuffer / TimeStruct`：承载带 payload、带延迟的控制信息
- `StallSignals`：承载同周期就需要生效的背压/阻塞决定

## 旧 tick 顺序与新 tick 顺序

在这个 PR 之前，CPU 的 tick 顺序基本是：

```text
fetch -> decode -> rename -> iew -> commit
```

这个 PR 之后，顺序改成：

```text
commit -> iew -> rename -> decode -> fetch
```

见：

- `src/cpu/o3/cpu.cc`

注意，这不表示“指令反着流动”。
真正变化的是：

- 软件模拟时，先让后级决定自己能不能接收
- 再让前级根据这个决定在同一拍是否继续推进

## 为什么这会影响 stall / block / unblock / squash

这是这个 PR 的本质所在。

### 旧模型的问题

旧顺序下：

- 前端先 tick
- 后端后 tick
- 如果后端这拍才发现资源满了，前端往往已经先做完了本拍动作

为了补这个时序差，代码里需要维护较复杂的状态机，例如：

- blocked
- unblocking
- squashing
- serialize stall
- skid buffer
- block/unblock 握手信号

这些逻辑能工作，但难读、难维护，也不利于 SMT 扩展。

### 新模型的改进

反向 tick 后，传播链会更直观：

1. `Commit` 先决定是否阻塞 `IEW`
2. `IEW` 立即决定是否阻塞 `Rename`
3. `Rename` 立即决定是否阻塞 `Decode`
4. `Decode` 立即决定是否阻塞 `Fetch`
5. `Fetch` 这拍就能看到最终反压结果

也就是：

```text
Commit -> IEW -> Rename -> Decode -> Fetch
```

这条链就是新的“同周期背压链”。

## 为什么这对 SMT 更友好

这个 PR 并没有完整实现 SMT 调度，但它在结构上更适合以后扩展 SMT。

原因主要有三点：

1. `StallSignals` 本身就是按线程数组组织的
2. 多个 stage 现在先把输入搬到本地 buffer，再决定消费多少，更适合做 per-thread arbitration
3. 背压链是显式的，不再过度依赖分散状态机

所以更准确地说，这个 PR 是：

- 不是“已经完成 SMT”
- 而是“把控制结构改成更适合后续 SMT”

所以它今天的作用不是“我没收走你就替我卡住值”，而是：

**“我负责把某个信息在 N 个周期后送到对面。”**

**2. 为什么它不像 RTL 里的级间寄存器那样天然保值**

你说得很对，RTL 里常见的是：

- 上一级 `valid`
- 下一级 `ready`
- `valid & ready` 才真正握手
- 没握手成功时，寄存器/FIFO 里的内容保持不变

而 gem5 这里不是这么统一建模的。

`TimeBuffer` 的实现本质上是一个“时间窗口”：

- `getWire(0)`：当前写槽
- `getWire(-delay)`：延迟若干拍后读槽
- 每拍末尾 `advance()`
- `advance()` 时会把新的未来槽清零重建

实现见 [timebuf.hh:179](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/timebuf.hh#L179)。

所以如果你问：

**“它会不会直接覆盖 timebuffer 中的值？”**

答案是：

**会，它会随着 `advance()` 轮转并清空未来槽位。**

但这不代表功能错误，因为在 gem5 这套设计里：

- **需要长期保留的数据，不应该只靠 `TimeBuffer` 存着**
- 应该已经被搬进某个 stage 的本地 buffer / 队列 / ROB / LSQ

这也是这次 PR 里大量出现 `fixedbuffer` 的原因。

**3. 那后级停收时，数据到底保存在谁那里**

这是这次 PR 最关键的理解点。

### RTL 直觉
RTL 里你会想：

- 下一级没 `ready`
- 那上一级寄存器里的 `valid+data` 继续保持

### gem5 这次 PR 的做法
它更像这样：

1. 上一级把“本拍准备发送的数据”写进级间 `TimeBuffer`
2. 下一级在自己的 `tick()` 里把这些数据**搬进自己的本地 buffer**
3. 真正是否继续处理，由下一级根据 `stallSig` 决定
4. 如果处理不完，就把数据继续留在**自己的本地 buffer**里，而不是要求 `TimeBuffer` 替它保值

例如：

- Decode 有 `fixedbuffer`

- Rename 有 `fixedbuffer`

所以更准确地说：

**gem5 这里把“链路延迟”和“接收端缓存”解耦了。**

**4. `StallSignals` 是不是就相当于握手信号**

可以说**很像“背压/ready”那一半**，但**不是完整的 `valid-ready` 协议**。

### 相似点
它的作用确实很像：

- 后级说“我现在收不了”
- 上级这一拍就别再往我这里推进

比如：

- Commit 决定 `blockIEW`
- IEW 决定 `blockRename`
- Rename 决定 `blockDecode`
- Decode 决定 `blockFetch`

这条链就是典型背压链。

### 不同点
但它不是标准 RTL `valid-ready` 的完整镜像，因为这里：

1. **`valid` 没有被统一显式建模成一个通道协议**
   - 是否“有数据”通常由 `fromX->size`、本地 queue/buffer 是否为空来表达
2. **`ready` 也不是一个统一字段**
   - 而是拆成了各级各自写的 `block*`
3. **数据保持不靠通道寄存器本身**
   - 而靠本地 `fixedbuffer`、ROB、LSQ 等结构

所以你可以这么记：

- `StallSignals` 更像 **“ready/allowin 的反向背压简化版”**
- 但不是“完整 valid-ready 通道抽象”

### `TimeBuffer` 解决的是

  - `combinational backpressure`
- 本质上只是把“两种机制”包进了一个类里

也就是：

**表面合并了，语义并没有真的统一。**

- valid
- delay
- backpressure mode
- buffering policy

1. **DelayedPayloadChannel**

   - 用于 squash/redirect/前向 bundle

2. **BackpressureChannel**

   - 用于同拍 allow/block

1. 代码概念更统一
2. 维护者更不容易把语义混掉

1. 要动大量 stage 代码
2. 现在很多逻辑是按 O3 历史写法累积的
3. 一旦抽象过头，反而更难 debug

- `Fetch`

  - 与分支预测器、FTQ、Icache 交互
- `Decode`

- `IEW`
  - 分发到 IQ / LSQ
  - 发射、执行、写回
- `Commit`

## 4. 旧模型与新模型的最重要区别

### 4.1 旧模型的 tick 顺序

在这个 PR 之前，CPU 的 tick 顺序基本是：

```text
fetch -> decode -> rename -> iew -> commit
```

这意味着：

- 前端先运行
- 后端后运行
- 如果后端在这一拍才发现资源满、需要 squash 或者需要阻塞，那么前端往往已经把这一拍干完了

因此旧模型需要维护很多额外状态来“补时序差”，例如：

- `Blocked`
- `Unblocking`
- `Squashing`
- `SerializeStall`
- `skidBuffer`
- block/unblock 握手信号

这些东西并不是无意义的，它们是在旧 tick 顺序下为了保持行为正确而产生的。

### 4.2 新模型的 tick 顺序

这个 PR 之后，tick 顺序改成：

```text
commit -> iew -> rename -> decode -> fetch
```

注意，这并不表示“指令反向流动”。

真正变化的是：

- 软件模拟时，后级先更新自己的状态
- 然后前级在同一拍根据后级状态决定是否推进

所以新模型下更自然的背压链是：

```text
Commit -> IEW -> Rename -> Decode -> Fetch
```

也就是：

1. `Commit` 先判断自己是否还能接收新的 rename 结果、ROB 是否还能插入
2. `IEW` 根据 commit 的反馈决定是否继续向前接收
3. `Rename` 根据 IEW/资源情况决定是否继续 rename
4. `Decode` 根据 rename 是否能继续推进来决定是否阻塞 fetch
5. `Fetch` 最后在这一拍就能知道自己能不能继续往前送

这就是所谓的 reverse ordered tick。

## 5. `TimeBuffer` 到底是什么

### 5.1 不要把它简单理解成“普通变量”

`TimeBuffer` 不是普通共享变量，它更像“带显式时间偏移的环形缓冲”。

关键语义如下：

- `getWire(0)`：访问当前时刻的槽位
- `getWire(-delay)`：读取延迟 `delay` 拍后才可见的槽位
- `advance()`：在周期边界推进整个时间窗口

因此，它的主要用途不是“简单共享一块数据”，而是**显式建模 stage 间传播延迟**。

实现见：

- `src/cpu/timebuf.hh`

### 5.2 `TimeBuffer` 不是 RTL 意义上的“自动保值级间寄存器”

这点非常重要。

在 RTL 里，我们通常会这样想：

- 上一级把 `valid + data` 放在级间寄存器或 FIFO 中
- 如果下一级 `ready=0` 没收走，那么这个寄存器/FIFO 里的值会继续保持

而在这个 gem5 O3 重构里，语义被拆开了：

1. `TimeBuffer`
   - 主要负责“这个信息跨多少拍可见”
   - 更像“带延迟的链路”或“时间总线”
2. 本地 buffer / 队列 / ROB / LSQ
   - 主要负责“这个数据如果本拍没被处理，要继续留着”
3. `StallSignals`
   - 主要负责“这一拍是否允许上游继续推进”

所以一句话记忆就是：

**`TimeBuffer` 更像延迟总线，而不是永远替你保值的流水寄存器。**

### 5.3 那 `TimeBuffer` 会不会覆盖旧值

会。

`TimeBuffer` 随着每拍 `advance()` 会推进时间窗口，并重建未来槽位。也就是说：

- 它的内容不是“只要没人消费就一直保留”
- 它本质上是按时间流动的

因此，**真正需要长期保留的数据，不能只依赖 `TimeBuffer` 保存**。

如果某一级已经收到数据但暂时处理不了，正确做法应该是：

- 先把数据搬入该 stage 的本地 buffer
- 然后由本地 buffer 持有这些数据直到后续周期继续处理

这也是这次 PR 中大量引入 `fixedbuffer` 的直接原因。

## 7. `StallSignals` 是什么

### 7.1 它的本质

`StallSignals` 是这次 PR 新增的“快速背压通道”。

它不是用来装复杂 payload 的，也不是用来表达多拍传播的，而是用来表达：

- 后级这拍能不能接收
- 上游这拍是不是该停

它是这条同拍背压链的载体：

```text
Commit -> IEW -> Rename -> Decode -> Fetch
```

### 7.2 为什么它存在

如果仍然让“阻塞信息”完全通过 `TimeBuffer` 传播，那么由于 `TimeBuffer` 是按拍延迟可见的，前级会晚一拍知道后级已经堵住。

一旦晚一拍，就又需要：

- block / unblock 状态机
- skid buffer
- 额外的中间状态

也就是说，`StallSignals` 的存在，就是为了把“是否准许推进”这类必须同拍生效的控制，从 `TimeBuffer` 中分离出来。

### 7.3 它是不是握手信号

可以说它**类似 RTL `valid-ready` 协议中 `ready/allowin` 的那一半**，但并不是完整的 `valid-ready` 通道抽象。

原因如下：

1. 这里没有一个统一的 `valid` 信号字段
   - 是否有数据，通常由 `size`、队列是否为空、本地 buffer 是否为空等方式表达
2. `ready` 也没有被统一抽象成一个接口
   - 而是拆成了 `blockFetch / blockDecode / blockRename / blockIEW`
3. 数据保持不靠通道本身完成
   - 而是靠各级自己的本地 buffer / ROB / LSQ 来完成

所以更准确地说：

- `StallSignals` 是“同拍背压”的建模机制
- 它与 RTL 中的 `ready/allowin` 很相似
- 但它不是完整 `valid-ready` 协议的一比一翻版

## 8. 与 RTL `valid-ready` / FIFO / 级间寄存器的对应关系

这一节最容易帮助硬件背景的开发者建立直觉。

### 8.1 RTL 中的典型模型

在 RTL 中，一个级间通道通常同时具备三层含义：

1. **数据通道**
   - `data`
2. **有效性**
   - `valid`
3. **可接收性**
   - `ready`

握手条件一般是：

```text
valid && ready
```

一旦握手成功：

- 数据向下一级推进

如果握手不成功：

- 数据保持在当前级间寄存器/FIFO 中
- 等待后续周期重试

### 8.2 新版 gem5 O3 中的对应关系

可以用下面这张表理解：

| RTL 概念 | 新版 gem5 O3 对应物 | 说明 |
| --- | --- | --- |
| 级间链路延迟 | `TimeBuffer` | 表达“几拍后可见” |
| 级间寄存器/FIFO 中的数据保持 | 各 stage 的 `fixedbuffer`、`fetchQueue`、`ROB`、`LSQ` 等 | 表达“已收到但暂未消费的数据继续保留” |
| `ready/allowin` 背压 | `StallSignals` | 表达“这一拍能不能继续往前推” |
| `valid` | `size`、队列非空、buffer 非空等隐式条件 | 没有统一字段，分散在具体结构里 |
| payload 控制信息 | `TimeStruct` / 前向 Struct | squash、redirect、doneSeqNum 等 |

### 8.3 一个更直观的类比

如果用硬件设计语言来类比，新模型更像下面这种结构：

```text
前向链路: 有延迟的数据通道
本地级内: 有输入缓冲 / FIFO / 寄存器保存未消费数据
反向链路: 有组合式 allowin / backpressure
```

而不是一个统一的“每一级都只靠一组 valid-ready 寄存器”的极简模型。

### 8.4 这和“寄存器保值”的关系

RTL 中你会说：

- 级间寄存器保值

新版 gem5 O3 中更合适的说法是：

- **接收级自己的本地 buffer 保值**

也就是说，不是 `TimeBuffer` 帮你 hold 住数据，而是：

- 下一级先把输入搬进自己的本地 buffer
- 如果这拍处理不完，本地 buffer 继续保留
- 上游则通过 `StallSignals` 被挡住，不再继续注入更多数据

这是和 RTL 直觉最像、但实现手法不同的地方。

## 9. 这次 PR 中各机制分别承担什么责任

### 9.1 `TimeBuffer` / `TimeStruct`

适合承载：

- 带 payload 的控制信息
- 明确需要跨拍延迟传播的信息
- stage 间前向传递的指令包

典型例子：

- squash
- redirect PC
- `doneSeqNum`
- `doneMemSeqNum`
- resolved CFI
- 各级向下一阶段发送的指令 bundle

### 9.2 本地 `fixedbuffer`

适合承载：

- 已经到达本 stage，但本拍还没处理完的数据
- 需要跨拍继续保留的输入

典型场景：

- Decode 已经收到了来自 Fetch 的若干指令，但这拍只能处理一部分
- Rename 已经收到了来自 Decode 的指令，但这拍资源不够
- IEW 已经收到了来自 Rename 的指令，但 IQ / LSQ / dispatch 带宽不足
- Commit 已经收到了来自 Rename 的指令，但这拍不一定能全部插入 ROB

### 9.3 `StallSignals`

适合承载：

- 这一拍是否允许上游继续推进
- 没有复杂 payload 的简单 backpressure

典型语义：

- `blockIEW`：commit 告诉 IEW，这一拍不要再继续向前推进到自己这里
- `blockRename`：IEW 告诉 rename，这一拍别再往下送
- `blockDecode`：rename 告诉 decode，这一拍别再往下送
- `blockFetch`：decode 告诉 fetch，这一拍别再继续出队/送指令

## 10. 为什么 `TimeBuffer` 和 `StallSignals` 不能简单视为同一种东西

这是后续维护最容易混淆的点。

### 10.1 它们解决的是两类不同问题

`TimeBuffer` 解决的是：

- 延迟传播
- 历史/未来时间槽位
- 带 payload 的信息移动

`StallSignals` 解决的是：

- 同拍背压
- 不需要等待 `advance()`
- 通常不带复杂 payload

### 10.2 为什么不能简单“全都塞进 `TimeBuffer`”

如果把同拍背压也塞回 `TimeBuffer`，会重新出现旧问题：

- 前级晚一拍知道后级阻塞
- 需要恢复 block/unblock 状态机来补时序差

### 10.3 为什么也不能简单“全都改成 `StallSignals`”

如果把 squash、redirect、doneSeqNum、payload 丰富的控制也都改成 `StallSignals`，会出现另一个问题：

- 这些信息本来就具有明确的多拍传播语义
- 且往往需要携带复杂数据
- 仅靠一个简单的 block 信号无法表达

因此两者在语义上是分工明确的。

## 11. 那有没有机会把两者“合并”

### 11.1 结论

**可以考虑做接口层统一抽象，但不建议简单把当前实现硬合并。**

### 11.2 为什么“真正合并”很难

因为语义上它们仍然是两类东西：

- 一类是 delayed payload channel
- 一类是 same-cycle backpressure

即使未来要统一成一个更抽象的 `Channel` 类，内部通常也仍然需要区分：

- 是否带延迟
- 是否需要 payload
- 是否负责 buffering
- 是否负责 backpressure

换句话说：

- **实现接口可以统一**
- **语义层面依然需要分层**

### 11.3 更合理的长期方向

如果后续 SMT 和通道复杂度继续上升，可以考虑做更高层的抽象，例如：

- `DelayedPayloadChannel`
- `BackpressureChannel`
- `BufferedStageInput`

但在当前阶段，更重要的是：

1. 先把语义边界写清楚
2. 先避免后续开发者误用
3. 再考虑是否值得抽象统一

## 12. 这一版设计为什么更适合 SMT

这个 PR 不是完整 SMT 支持，但它在结构上更适合未来做 SMT。

原因包括：

1. `StallSignals` 本身就是按线程数组组织的
2. 多个 stage 先把输入搬入本地 buffer，再决定消费多少，这更接近 per-thread arbitration 的思路
3. 背压链显式化后，更容易按线程分析“是哪个下游线程/资源导致了阻塞”

因此更准确地说：

- 这不是“SMT 已经实现完成”
- 而是“流水线控制结构更适合后续 SMT 开发”

## 13. 一次典型周期里会发生什么

下面用一个典型例子说明新模型的直觉。

假设这一拍后端资源紧张，Commit 或 IEW 已经接近满载。

### 周期内顺序

1. `Commit` 先 tick
   - 判断 ROB 是否还能接收、是否正在 squashing
   - 设置 `blockIEW`
2. `IEW` 再 tick
   - 读取 `blockIEW`
   - 决定自己是否还能继续从 Rename 接收
   - 必要时设置 `blockRename`
3. `Rename` 再 tick
   - 读取 `blockRename`
   - 决定这拍是否继续 rename
   - 必要时设置 `blockDecode`
4. `Decode` 再 tick
   - 读取 `blockDecode`
   - 决定是否阻塞 Fetch
   - 设置 `blockFetch`
5. `Fetch` 最后 tick
   - 读取 `blockFetch`
   - 决定是否继续向 Decode 发送指令
6. 周期末统一 `advance()` 各类 `TimeBuffer`

### 需要注意的点

在这个过程中：

- 指令本身的前向传播，仍然依赖前向 `TimeBuffer`
- 某一级已经收到但本拍处理不完的数据，会保存在该级本地 buffer 中
- 是否允许上游继续推进，则由 `StallSignals` 同拍决定

## 14. 开发者修改代码时的判断准则

后续如果要继续修改这套流水线代码，建议始终按下面的准则判断。

### 14.1 什么时候应该用 `StallSignals`

当你想表达的是：

- “这一拍我能不能接收”
- “上游这一拍应不应该继续推”
- “这个决策需要同拍生效”

此时优先考虑 `StallSignals`。

### 14.2 什么时候应该用 `TimeBuffer`

当你想表达的是：

- “这个信息需要延迟若干拍后可见”
- “这个信息不是简单 bool，而是带 payload”
- “这是一个控制事件，而不是简单 backpressure”

此时优先考虑 `TimeBuffer` / `TimeStruct`。

### 14.3 什么时候应该放进本地 buffer

当你想表达的是：

- “数据已经到达本级”
- “但这拍没处理完”
- “需要跨拍继续保留”

此时应该放进本级的 `fixedbuffer` 或其他局部队列/结构，而不是指望 `TimeBuffer` 自动保值。

## 15. 后续维护最容易踩的坑

### 15.1 把“应当同拍生效”的信号误放进 `TimeBuffer`

后果：

- 信号晚一拍
- 前级多推进一拍
- 又要重新引入复杂 block/unblock 补偿逻辑

### 15.2 把“应当带 payload/带延迟”的控制误简化成 `StallSignals`

后果：

- squash / redirect / doneSeq 等复杂语义表达不完整
- 时序语义变得含糊

### 15.3 误以为 `TimeBuffer` 会自动替 stage 持有未消费数据

后果：

- 数据保留逻辑写错
- 修改后很容易出现“本拍看起来没问题、下拍数据没了”的问题

### 15.4 把本地 buffer 和上游背压脱钩

如果一个 stage 已经出现局部积压，但没有及时通过 `StallSignals` 阻止上游继续输入，就容易出现：

- 带宽模型失真
- buffer overflow
- 行为与真实流水线不一致

### 3.4 `fetchStallReason`

用途：

## 4. 为什么这不是重复统计

表面看起来四级都在记同一个 reason，好像重复。

举个例子：

假设真正根因是 `LoadL2Bound`。

那可能会出现这种链条：

1. IEW/dispatch 因为 load miss 卡住，记录 `dispatchStallReason=LoadL2Bound`
2. Rename 这拍因为 IEW 不收了，也记 `renameStallReason=LoadL2Bound`
3. Decode 因为 rename 不收了，也记 `decodeStallReason=LoadL2Bound`
4. Fetch 因为 decode 不收了，也记 `fetchStallReason=LoadL2Bound`

这四个统计**不是重复**，而是在说：

- 这个 L2 bound 问题不仅出现在 dispatch
- 它还把整条前端链都压住了

- `ITlbStall`
- `FTQBubble`

- `LoadL2Bound`
- `ScalarLongExecute`
- `SerializeStall`
- `MemCommitRateLimit`

这说明前端没送满只是结果，不是原因。

这两类情况在优化方向上完全不同。

所以如果 `fetchStallReason` 按口径 B 修好了，它最大的价值就是：

**帮助区分“前端没送满”究竟是 frontend 自己差，还是后端把前端憋住了。**

这在 topdown 里其实很关键。
