**结论**

**背景**

GitHub Actions 的粒度是“触发 workflow”，不是“从 UI 里任选某个 job”。  
也就是说：

- `workflow_dispatch` 只能启动整个 workflow。
- 如果想“只跑某一个 job”，必须在 YAML 里自己加选择逻辑。
- 你这个文件里最后一个 job 本身是**自包含**的，没有 `needs` 依赖前面的 job，所以改起来其实很容易。

**比较好的办法**

**方案 1：单独拆一个 workflow，最干净，也最推荐**

给 `raw_linux_boot_test` 单独建一个文件，比如：

- [`.github/workflows/raw-linux-boot.yml`](/nfs/home/yanyue/workspace/GEM5_review/.github/workflows)

这样你手动触发时，直接选这个 workflow，就只跑 raw linux boot，不会把大套餐一起带上。

优点：

- UI 最直观
- 不污染现有 `gem5.yml`
- 不需要给每个 job 都写 `if`
- 这个 job 本来就独立，拆出去几乎没有结构风险

缺点：

- 会有一点 YAML 重复

如果后面重复太多，再把通用 build 步骤抽到 reusable workflow 或 composite action 即可。

**方案 2：保留在同一个文件里，加一个手动输入参数**

比如把 [`workflow_dispatch`](/nfs/home/yanyue/workspace/GEM5_review/.github/workflows/gem5.yml#L8) 改成这样：

```yaml
on:
  push:
    branches: [ xs-dev ]
  workflow_dispatch:
    inputs:
      target:
        description: "Which job to run"
        required: true
        default: "all"
        type: choice
        options:
          - all
          - raw_linux_boot_test
```

然后给各个 job 加 `if`。

前面的 job：

```yaml
if: ${{ github.event_name != 'workflow_dispatch' || github.event.inputs.target == 'all' }}
```

最后这个 `raw_linux_boot_test`：

```yaml
if: ${{ github.event_name != 'workflow_dispatch' || github.event.inputs.target == 'all' || github.event.inputs.target == 'raw_linux_boot_test' }}
```

这样效果是：

- `push` 到 `xs-dev` 时，仍然全部照常跑
- 手动触发时，可以在 UI 里选 `all` 或 `raw_linux_boot_test`

**我更推荐哪个**

我更推荐**方案 1：拆成单独 workflow**。  
理由不是“更高级”，而是**更省心**：

- 你现在的诉求非常具体，就是“偶尔只想手动跑最后一个”
- 最后一个 job 本身独立
- 这个仓库本来就已经有不少 workflow 文件，不差再多一个

如果你改成方案 2，以后每加一个新 job，都得顺手维护 `inputs` 和 `if`，长期会变乱。

**风险点**

- 如果你拆成单独 workflow，后续有人修改了 `raw_linux_boot_test` 的命令，但忘了同步两个文件，可能会漂移。
