**结论**
更像“结构竞争”，不是“纯语义难预测”。

原因很直接：
- 两个 workload 的热点在 `run381` 和 `run382` 中都稳定落在同一小撮函数和同一批 32B/64B block。
- 多个最热 block 内同时塞了 2 到 7 条热点 branch，且这些 branch 往往属于同一个紧凑循环或同一个局部判定链。
- `nngs` 的 `compute_connection_distances`、`trevord` 的 `order_moves`/`chainlinks2` 都呈现出典型的 branch-dense 局部块形态，支持“局部块内竞争/alias/容量压力”判断。

保留意见：
- `remove_liberty`、`remove_neighbor`、`fastlib` 这类点仍有数据相关的语义难度。
- 但从“热点是否聚集在少数 code block/函数里”这个问题本身看，证据明显偏向结构竞争。
