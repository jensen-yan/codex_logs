**结论**

你现在最该做的，不是先去重编 Linux，而是先把“它到底是没前进、还是前进了但没串口输出、还是在 MMIO/中断轮询里活锁”这三件事分开。

我先把几个关键事实钉住：

- `kmhv3.py` 默认会打开 difftest，不需要你显式加 `--enable-difftest`。见 [xiangshan.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/xiangshan.py#L718) 和 [xiangshan.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/xiangshan.py#L140)。
- 你的 `--raw-cpt --generic-rv-cpt linux.bin` 不是“像 ELF 那样加载 kernel”，而是把 raw binary 直接 mmap 到物理内存里跑，reset vector 是 `0x80000000`。见 [fs_workload.cc](/nfs/home/yanyue/workspace/GEM5_review/src/arch/riscv/bare_metal/fs_workload.cc#L60) 和 [physical.cc](/nfs/home/yanyue/workspace/GEM5_review/src/mem/physical.cc#L550)。
- 我直接对 `linux.bin` 跑了 `strings`，里面同时有 `OpenSBI` 和 `Linux version 6.1.83`。所以 `Boot HART MEDELEG` 更像是 OpenSBI/boot 阶段打印，不一定已经进 kernel 了。只重编 Linux 内核本体，未必能打到你要的点。
- difftest 不等于“死锁检测器”。尤其这个实现里，MMIO 指令会直接跳过 NEMU 单步，见 [base.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/base.cc#L902)。如果 guest 卡在设备轮询里，完全可能“没报 diff，但一直跑不出来”。

还有一个非常大的嫌疑点：

- GEM5 这边的 `NemuPlic` 基本是桩实现，读永远返回 0，写也不做实事。见 [nemu_plic.cc](/nfs/home/yanyue/workspace/GEM5_review/src/dev/riscv/nemu_plic.cc#L16)。
- 而 `makeBareMetalXiangshanSystem()` 这套平台本身也不是你 QEMU `-M nemu` 的原样复刻，它挂的是 `UartLite + Clint + NemuMMC + NemuPlic`。见 [FSConfig.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/FSConfig.py#L678)。

- 真的卡死在设备/中断/CSR/MMU 路径。
- 实际还在跑，但 kernel 后续串口路径和平台/DTB 不匹配，导致“最后一条可见输出停在 OpenSBI”。

**当前结论**

我已经把第一个明确的 gem5 问题修掉了：gem5 在没开 `Sv48` 时，仍然接受了 `satp.mode=Sv48`。我在 [isa.cc](/nfs/home/yanyue/workspace/GEM5_review/src/arch/riscv/isa.cc#L577) 和 [isa.cc](/nfs/home/yanyue/workspace/GEM5_review/src/arch/riscv/isa.cc#L713) 把 `SATP/VSATP` 写入都改成了受 `enableSv48` 约束。重编后，启动已经从只停在 OpenSBI 推进到了真正进入 Linux early boot。

现在新的真实 blocker 不是 difftest，也不是 `linux.bin` 本身加载失败，而是 Linux 在 very early paging 阶段 panic。实际抓到的关键信息是：

```text
Unable to handle kernel paging request at virtual address ffffffc6febfe000
epc : 0xffffffff800d531c
ra  : 0xffffffff800dc9c4
cause: 0xf
```

我把这串地址反汇编了，`0xffffffff800d531c` 落在 `memset` 的 4KB 清零循环里，`0xffffffff800dc9c4` 的上一条正好是在调用这个 `memset(…, 0, 0x1000)`。结合后面的回溯地址链，我的判断是：

- 这是 Linux 在建立最终页表时，给“新分配的一页页表页”做清零。
- `badaddr=ffffffc6febfe000` 很像落在 RISC-V Sv39 的 `fixmap` 区域。
- 所以更像是“Linux 刚建立了一个临时/fixmap 映射，随后立刻访问它；gem5 没把这个映射及时看见”，而不是 Linux 自己随便写飞了。

这部分是基于回溯和反汇编的**推断**，但强度很高。它比“重编一份 linux.bin 再加 printk”更靠近根因。

**我已经确认过的几点**

- difftest 关掉后，问题仍然存在，所以当前不是 NEMU diff 引入的。
- payload 内嵌 DTB 的内存 `reg` 实际是 `base=0x80000000, size=8GB`，和当前 gem5 配置一致；`memory@100000000` 只是节点名看起来怪，不是根因。

**最新结论**

问题又缩小了一大步，而且现在不是泛泛的“MMU 有问题”了。

我已经确认：

- fault 不是发生在根页表或中间页表阶段。
- walker 已经走到 **最后一级 L0 页表**。
- 真正缺失的叶子 PTE 槽位是 **物理地址 `0x803b3ff0`**。
- fault 时对应的整条 64B PTE line 是从 **`0x803b3fc0`** 读出来的，而且当时是全 0。

这次抓到的关键日志是：

```text
PTW probe: level=0 walk_addr=0x803b3fc0 entry_vaddr=0xffffffc6febfe0b8 idx=0x6 pte=0 ...
Walker invalid PTE before write fault:
  pc=0xffffffff800d5378
  req_vaddr=0xffffffc6febfe0b8

  read_paddr=0x803b3fc0
  level=0 choose=0x6
  pte=0
  line=[0 0 0 0 0 0 0 0]
```

`idx=6` 对应的槽位就是 `0x803b3fc0 + 6*8 = 0x803b3ff0`。

**已经排除/验证过的事**

- 我之前试过把 `sfence_vma` 当成 barrier，并在 O3 `commit` 里让 barrier 头指令也先 `flushAllStores()`。
- 这个修改**没有绕过当前 panic**，Linux 还是死在同一个地址 `ffffffc6febfe000`，同一个 `epc=0xffffffff800d531c`。
- 所以根因**不太像只是**“`sfence.vma` 没把前序 store 排空”。

**当前最可能的两类根因**

1. Linux 本应写入 `0x803b3ff0`，但 gem5 的 store 路径根本没把这笔写发出去。
2. 这笔写发出过，但 walker 在 fault 那一刻仍然没看到它。

这两类现在已经可以用一个很窄的 watchpoint 区分。
