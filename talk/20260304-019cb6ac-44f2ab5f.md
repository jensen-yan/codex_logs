**源码语义**
**1. `0x3b23a`: `do_play_move()`**
对应 [board.c#L3399](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L3399) 一带。

它本质上在做：

```c
if (UNMARKED_COLOR_STRING(west, color)) { ... }
else if (UNMARKED_COLOR_STRING(west, other)) { ... }
```

也就是判断西侧邻居是：

- 本方串
- 对方串
- 是否已 mark

这是局部棋盘内容驱动，不是迭代号驱动。  
所以这条分支如果 IMLI 有帮助，通常不是因为它“是 loop 分支”，而更像：

- 外层 reading phase 被 IMLI 区分开了
- 同一个 PC 的上下文 alias 减少了

**2. `0x3fa08`: `fastlib()`**
对应 [board.c#L1429](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1429)。

它是在四邻域分类链入口：

```c
if (board[SOUTH(pos)] == color) { ... }
else if (board[WEST(pos)] == color) { ... }
else if (board[NORTH(pos)] == color) { ... }
else if (board[EAST(pos)] == color) ...
```

也是棋型谓词分支。  
它的 taken/not-taken 主要由当前位置周围棋子颜色决定，不是 loop 迭代计数。

**3. `0x441a2`: `special_rescue_moves()`**
函数体在 [reading.c#L1692](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L1692)，但真正对应到的是 `ADD_CANDIDATE_MOVE` 宏里的去重扫描，宏在 [reading.c#L38](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L38)。

也就是：

```c
for (u = 0; u < moves->num; u++)
  if (moves->pos[u] == move) ...
```

这类分支的结构是：

- 前几次迭代几乎都继续扫
- 命中重复项或者扫到尾部才翻转

这正是 IMLI 擅长建模的模式。

**4. `0x4d04c`: `order_moves()`**
源码在 [reading.c#L4156](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L4156)，具体是 selection sort 内层：

```c
for (j = i + 1; j < moves->num; j++) {
  if (moves->score[j] > maxscore) {
    maxscore = moves->score[j];
    max_at = j;
  }
}
```

`0x4d04c` 就是这个内层 `j` 循环的回边。  
这是最干净的 IMLI 样本之一，因为它几乎就是：

- 连续若干次 taken
- 最后一次 not-taken

而且外层 `i` 还会改变内层 trip count。  
没有 iteration-aware 历史时，非常容易 alias。

**5. `0x40d9e`: `chainlinks2()`**
对应 [board.c#L1050](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1050) 左右的链扫描逻辑。  
它也是典型扫描回边，和 `0x4d04c/0x441a2` 属于一类，只是语义上不是排序，而是邻接链枚举。

---

**为什么 IMLI 对 gobmk 有效果**
可以总结成一句话：

`gobmk` 里有一批热点分支，本质上不是“棋局本身难”，而是“同一个 PC 在短小但高频的扫描/排序循环中重复出现，最后一次翻转很依赖当前是第几次迭代”。这类模式天然适合 IMLI。

更具体地说：

- `special_rescue_moves` 的去重扫描
- `order_moves` 的 selection sort 内层
- `chainlinks2` 的邻接枚举

这些都具有共同结构：

- 同一静态 PC 连续执行多次
- 大部分动态实例行为一致
- 只有接近尾部时才翻转
- 不同调用下 trip count 又不同

普通 global history 往往只能看到：

- “这个 PC 经常 taken，偶尔 not-taken”

IMLI 能额外编码：

- “这是第 1 次 / 第 2 次 / 倒数第 1 次”

所以它对这类 PC 生效是完全合理的。

**为什么 gem5 有效，但 RTL 没效果**
我认为你现在最该优先怀疑的，不是“RTL 的 I 表容量不够”，而是**语义不一致**。

当前 gem5 里的 IMLI 实现并不是一个通用 loop detector，而更像：

- 基于 `backward-taken streak` 的计数器
- 且它吃的是 fetch block 里“第一个 taken cond branch”的 backward/taken 信息

相关位置：

- [common.hh#L606](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/common.hh#L606)
- [folded_hist.cc#L149](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/folded_hist.cc#L149)
- [folded_hist.cc#L156](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/folded_hist.cc#L156)
- [btb_mgsc.cc#L1129](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L1129)

这意味着：

- gem5 的 IMLI 未必真的是“目标 branch 的 loop iteration”
- 它可能是一个代理信号
- 但这个代理信号在 gobmk 的这些热点上恰好有用

如果 RTL 那边：

- IMLI 的定义不是 backward-taken streak
- 或不是按 gem5 这种 speculative update / recover 语义维护
- 或取的不是同一条 branch/同一 fetch block phase

那即使都叫 `IMLI`，行为也可能完全不同。

**一套可复用的方法论**
你说想总结一套能帮助 RTL 看波形、调试 IMLI 的方法。我建议固定按下面四步做。

**1. 先筛 branch，不要一上来就盯所有热点**
优先挑两类：

- `loop-exit / scan-backedge`
- `dirMiss` 高、target miss 近乎 0

像这次 gobmk 里最适合拿来做 IMLI case study 的就是：

- `0x4d04c`
- `0x441a2`
- `0x40d9e`

而不是先盯 `0x3b23a`

**2. 先定语义，再谈 predictor**
每个热点先问：

- 它是循环回边，还是棋型谓词？
- 是结构上应该好预测，还是语义本来就难？

如果是 `order_moves` 这种 selection sort 内层回边，预测差就更像 predictor 没学好。  
如果是 `do_play_move` 这种局部棋盘谓词，预测差不一定是 IMLI 的锅。

**3. RTL 看波形时固定看四个窗口**
按时间顺序：

- predict/lookup
- speculative history update
- squash/recover
- commit/update

最关键的 shortlist：

- `branchPC`
- `startPC`
- `pred_taken`
- `actual_taken`
- `pred_bw_taken`
- `real_bw_taken`
- `imli_counter_before`
- `imli_counter_after_spec`
- `imli_counter_after_recover`
- `iIndex`
- `iTable_raddr/rdata`
- `iTable_waddr/wdata/wen`
- `tage_conf`
- `totalSum`
- `totalThres`
- `useSc`

**4. 用判别树分问题类型**
如果 RTL 没效果，按这个顺序切：

- `iPercsum` 基本没方向感：没学到
- `iPercsum` 有方向但 `useSc` 少：被 gate 挡掉
- `iIndex` 看起来对，但值抖得厉害：alias
- squash 后 phase 明显漂：恢复/更新时间错误

**我对 gem5 当前 IMLI 的几个保留意见**
你问当前模型还有没有提升空间，我觉得是有的，而且不小。

1. `weight scaling` 实际没生效  
[btb_mgsc.cc#L301](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L301) 直接 `return percsum`，但 weight table 还在训练。  
这意味着当前 `iWeightTable` 对最终决策没真正贡献。

2. IMLI 语义过窄  
只看 backward-taken streak，不是通用 loop iteration。  
对 gobmk 这种复杂 reading phase，可能只抓住了“近似 loop 信息”。

3. 只有 1 个 I table  
默认 `iTableNum=1, iHistLen=[8]`，见 [BranchPredictor.py#L1110](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/BranchPredictor.py#L1110)。  
对多 phase 的 gobmk，单表很容易不够。

4. 没有 tag，alias 风险高  
当前是纯 counter array。  
如果你真要长期优化 gobmk，我会把“更强 hash / 轻量 tag”列进高优先级。

5. PC threshold 默认关  
对少数特别适合 IMLI 的 branch，缺少 per-PC gate 细化。

6. 更新策略偏保守  
[btb_mgsc.cc#L795](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L795) 只有“错了”或“margin 小”才训练。  
对稳定但稀疏的 loop-exit 分支，可能收敛偏慢。
