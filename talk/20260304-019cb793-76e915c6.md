**结论总览**

这 4 个热点 branch PC 都属于 `gobmk` 主体 ELF，而不是 `bbl/runtime`。依据是 [gobmk ELF](/nfs/share/zyy/spec06_rv64gcb_O3_20m_gcc12.2.0-intFpcOff-jeMalloc/elf/gobmk_base.riscv64-linux-gnu-gcc12.2.0) 的可执行 `LOAD` 段覆盖 `0x10000..0x1e7bd5`，而 `0x3b23a / 0x3fa08 / 0x441a2 / 0x4d04c` 都落在这个范围内。

按语义分类，我的判断是：

| PC | 函数 | 更像哪类 | 结论 |
|---|---|---|---|
| `0x3b23a` | `do_play_move` | `predicate-result` | 检查相邻点是否是本方未标记串，强烈依赖局部棋盘状态 |
| `0x3fa08` | `fastlib` | `predicate-result` | 邻居颜色分类的入口分支，决定后续“0/1/2 ally string”路径 |
| `0x441a2` | `special_rescue_moves` | `loop-exit` | 实际是 `ADD_CANDIDATE_MOVE` 宏里的去重扫描回边 |
| `0x4d04c` | `order_moves.lto_priv.0` | `loop-exit` | `selection sort` 内层扫描回边 |

这 4 个里，**最像 IMLI 甜点的是 `0x441a2` 和 `0x4d04c`**。  
原因很直接：它们的结果主要由“当前是这次循环的第几次迭代”决定，而不是纯粹由棋盘内容决定。IMLI 正好擅长给同一个 PC 附加 loop iteration phase 信息，区分“还在中间迭代”与“快到退出点了”。

```c
if (UNMARKED_COLOR_STRING(west, color)) { ... }
else if (UNMARKED_COLOR_STRING(west, other)) { ... }
```

**语义判断**

这是典型 `predicate-result`，不是 loop-exit。  
它在判断“西侧邻居当前属于哪种局部棋形状态”：

- 是不是本方串
- 是不是对方串
- 这串有没有在本次 `string_mark` 中被处理过

结果由**局部棋盘内容 + string mark** 决定，属于数据/状态谓词，不是简单迭代计数。

**为什么可能从 IMLI 获益**

这里只能说**可能间接受益，不是 IMLI 最对口的类型**。

原因：

- 这个分支本身不是 loop backedge；
- 它的难点主要来自局部棋盘状态，而不是 trip count；
- 如果 IMLI 有收益，更像是它帮助区分外层 reading/search 中不同 phase，减轻同一 PC 在不同局面上的 alias。

所以我更倾向于说：  
**这条分支若表现差，更像“谓词语义本身偏复杂/上下文混合重”，而不是“缺少 loop iteration 信息”这一个因素。**

```asm
...
3fa08: beq a7, a1, 0x3fc82
```

即先取 `board[pos + 0x14]`，再比较它是否等于 `color`，这正是四邻域颜色分类链的入口。

**源码语义对应**

对应 [board.c#L1441](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1441) 开始的邻居分类逻辑：

- [board.c#L1441](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1441)
- [board.c#L1457](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1457)
- [board.c#L1465](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/board.c#L1465)

核心语义是：

```c
if (board[SOUTH(pos)] == color) { ... }
else if (board[WEST(pos)] == color) { ... }
else if (board[NORTH(pos)] == color) { ... }
else if (board[EAST(pos)] == color) ...
```

它在确定落子点周围是否存在 0/1/2 个 allied string，以及是否需要 `return -1`。

**语义判断**

这条也更像 `predicate-result`。

原因：

- 它是对局部 board cell 的颜色判断；
- 后续路径会根据邻接串组合进入不同 case；
- 分支结果直接取决于当前位置邻域结构，不是规则 loop 模式。

严格说，它还有一点“局部状态分类器”的味道，但还不到我会叫它 `state-machine` 的程度。  
我更愿意把它归为**数据驱动的谓词分支**。

**为什么可能从 IMLI 获益**

同样是**间接受益多，直接受益少**。

可能的收益路径：

- `fastlib()` 在 tactical reading 中被非常频繁调用；
- 同一 PC 会在一些重复的 search phase 中被成批触发；
- IMLI 若给这类 phase 加了额外区分，可能减少 alias。

但本质上，这条分支是否 taken 主要由棋盘局部 pattern 决定。  
所以如果它是热点且难预测，我不会先怀疑“少了 IMLI”，而会先怀疑：

- 同一 PC 混入了太多不同局面；
- 历史建模不够表达局部结构；
- predictor 容量/alias 问题。

### 3. `0x441a2` -> `special_rescue_moves()` 内 `ADD_CANDIDATE_MOVE` 去重扫描 -> `loop-exit`

**符号映射**

- `llvm-symbolizer`: `special_rescue_moves`
- 函数在 [reading.c#L1692](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L1692)

**为什么不是表面那段 `for (k = 0; k < 4; k++)`**

单看源码函数体，[reading.c#L1698](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L1698) 是 4 邻居循环，看起来不该出现 `moves->num` 扫描。  
但这里调用了宏：

- [reading.c#L1711](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L1711)
- 宏定义在 [reading.c#L38](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L38)

宏展开后会先做去重扫描：

```c
for (u = 0; u < (moves).num; u++)
  if ((moves).pos[u] == (move)) { ... }
```

**汇编依据**

局部反汇编：

```asm
4418c: lw  a3, 0x190(s6)     // moves->num
...
4419e: lw  a4, 0(a5)         // moves->pos[u]
441a0: addi a5, a5, 0x4
441a2: bne a4, s2, 0x44198   // != candidate -> continue scan
441a6: bne a0, a3, 0x44154   // found duplicate before end
```

这和宏完全吻合。

**语义判断**

这条我归为 `loop-exit`。

它是一个非常典型的“线性搜索回边”：

- 还没找到重复项，就继续扫下一个元素；
- 找到重复项，提前退出；
- 扫到末尾，也退出并追加新候选。

**为什么这类分支会从 IMLI 获益**

这是本组里最标准的 IMLI 受益类型之一。

原因：

- 同一个 PC 在一次调用里会被重复执行多次；
- 大多数迭代上它会 taken，直到命中重复项或扫到尾部才翻转；
- 结果强依赖“当前是扫描的第几项”，这正是 loop iteration 信息；
- 普通 global history 容易把“同一 PC 的第 1 次、第 5 次、第 12 次访问”混在一起；
- IMLI 可以把这些访问按 iteration phase 区分开，尤其有助于学习“最后一次不再回跳”或者“命中重复项时提前停”的模式。

简化地说：

- **没有 IMLI**：只看到“这个 PC 最近经常 taken，但偶尔不 taken”
- **有 IMLI**：能更好地区分“现在还在扫描前半段”还是“已经接近扫描尾部/命中点”

所以这类 branch 的收益机制很清楚，且理由很硬。

### 4. `0x4d04c` -> `order_moves()` selection sort 内层回边 -> `loop-exit`

**符号映射**

- `llvm-symbolizer`: `order_moves.lto_priv.0`
- 源函数在 [reading.c#L4156](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L4156)

**源码语义对应**

最直接对应 [reading.c#L4327](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L4327) 到 [reading.c#L4336](/nfs/home/yanyue/tools/cpu2006_analyze/benchspec/CPU2006/445.gobmk/src/engine/reading.c#L4336)：

```c
for (j = i + 1; j < moves->num; j++) {
  if (moves->score[j] > maxscore) {
    maxscore = moves->score[j];
    max_at = j;
  }
}
```

**汇编依据**

局部反汇编：

```asm
4d03e: lw   a3, 0(a4)
4d040: bge  a2, a3, 0x4d048
4d044: mv   a0, a5
4d046: mv   a2, a3
4d048: addiw a5, a5, 0x1
4d04a: addi a4, a4, 0x4
4d04c: bne  a5, a1, 0x4d03e
```

`0x4d04c` 明显就是内层 `j` 循环的回边。

**语义判断**

这是最纯的 `loop-exit`。

它几乎不承载业务语义，主要承载的是：

- 内层扫描是否结束；
- 这轮还要不要继续找更高分 move。

**为什么这类分支会从 IMLI 获益**

这也是非常典型的 IMLI 适配对象。

原因：

- 对每个外层 `i`，内层 `j` 的 trip count = `moves->num - i - 1`；
- `moves->num` 很小但变化频繁，且不同战术局面下不同；
- 同一 PC 呈现“连续 taken 若干次，最后一次 not-taken”的结构化模式；
- 这个“最后一次”由当前迭代编号决定，而不是由最近几位全局分支历史决定。

IMLI 的价值就在于：

- 让 predictor 区分“这是本轮内层扫描的第 1 次”还是“最后 1 次”；
- 降低不同 `moves->num`、不同外层 `i` 造成的模式混叠；
- 对这种小而频繁的 O(n²) 排序回边尤其有效。

如果你专门在看“为什么老 gcc12 profile 里某些 gobmk 热点能吃到 IMLI 红利”，这条和 `0x441a2` 是最值得优先解释的。

---

**整体判断**

如果只按“像不像 IMLI 受益分支”排序，我会这样排：

1. `0x4d04c`  
2. `0x441a2`  
3. `0x3fa08`  
4. `0x3b23a`

前两条是**直接的 loop-iteration-sensitive branch**。  
后两条是**数据驱动 predicate branch**，即使有收益，也更像 IMLI 通过 phase disambiguation 间接帮忙，而不是正中其核心建模对象。

换句话说：

- `0x441a2` / `0x4d04c`：更像“IMLI 真能学到它们”
- `0x3b23a` / `0x3fa08`：更像“如果有收益，也不是因为它们本质上是 loop 分支，而是因为 IMLI 顺带改善了上下文区分”
