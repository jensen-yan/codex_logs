**前提**

我这次按 `mgsc-table-probe` 和 `branch-predictability-triage` 的思路读了仓库实现。仓库里我没有直接搜到现成的 `gobmk_nngs_5856` 目录，只搜到一个相近 case `[debug/gobmk_nngs_18098](/nfs/home/yanyue/workspace/GEM5_review/debug/gobmk_nngs_18098)`；所以下面给的是一套“面向 gobmk 这类 loop-exit / iteration-phase 分支”的 RTL 调试方法论，重点基于当前 gem5 代码，而不是假设 RTL 和论文版 IMLI 完全一致。

先给结论：这个仓库里的 IMLI 比很多 RTL 里的 loop-iter 机制更“简化”。它本质上不是显式 loop detector，而是“连续 backward-taken 次数计数器”，`taken` 就加一，`not taken` 就清零，见 [folded_hist.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/folded_hist.cc#L149)；而且它吃的输入不是“任意目标分支的迭代号”，而是 fetch block 里“第一个 taken 条件分支是否 backward”的结果，见 [common.hh](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/common.hh#L606) 和 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L1129)。这恰好是 gem5 有效、RTL 无效时最该优先怀疑的语义差异。

**1. 这类分支应该看哪些历史 / iteration 信号**

对 `gobmk_nngs_5856` 这类“同一静态分支在 loop 尾部翻转”的分支，我建议至少看 6 组信号：

- `branchPC` 本身的动态 outcome 序列，先确认是不是典型的固定 trip-count / near-tail flip。仓库单测里把 IMLI 典型模式直接建成了 `T,T,T,N`，并验证 ITable 能学到，见 [btb_mgsc.test.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/test/btb_mgsc.test.cc#L644) 和 [btb_mgsc.test.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/test/btb_mgsc.test.cc#L910)。

- `backward-taken` 事件流，不是一般的 taken/not-taken。当前 gem5 的 IMLI 只在 backward taken 时自增，否则清零，见 [folded_hist.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/folded_hist.cc#L156)。所以如果 RTL 的 loop-iter 信号来自 loop detector，而 gem5 来自 backward-taken streak，两边即使都叫 IMLI，也可能完全不是同一个 phase。

- `连续 backward-taken 次数`，也就是当前 `indexIFoldedHist[i]._folded` 对应的语义值。对 gobmk 这类分支，最关键的不是“历史长不长”，而是“第几次迭代”。在当前实现里，`iIndex` 就是 `startPC XOR folded_imli_counter`，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L366)。

- `TAGE confidence bucket`。MGSC 是否有机会覆盖，不只取决于 IMLI 学没学到，还取决于 TAGE 置信度是 high/mid/low，见 [btb_tage.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_tage.cc#L310) 和 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L420)。

- `fetch block startPC` 和目标分支在 block 内的位置。MGSC 的表索引用的是 `startPC` 加 folded hist，但 lane 选择又用了 `branchPC` 低位，见 [btb_mgsc.hh](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.hh#L273)。所以“同一分支跨 block 组织方式变化”会直接影响 IMLI 命中与 alias。

- 外层上下文信号。若 gobmk 这个分支的翻转点随外层 phase 漂移，只看 IMLI 可能不够，还要看 G/P/L 三类上下文贡献。当前 trace 会把 `bw/l/i/g/p/bias` percsum 全部打出来，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L222)。

gem5 的决策点就在 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L383) 到 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L437)。

2. `speculative history update` 窗口  
看预测之后 IMLI/ BW / G / P / L 是否一起按“预测结果”前推。gem5 在最终预测形成后，会先更新 folded hist，再更新 `s0History/s0BwHistory/s0PHistory/s0LHistory`，见 [decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/decoupled_bpred.cc#L906)。  
对 IMLI 最关键的是 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L1129)：它直接拿 `getBwHistInfo()` 的结果去 spec update。

3. `squash/recover` 窗口  
这是 RTL 最容易和 gem5 分叉的地方。看 mispredict 后是否先 restore 旧 snapshot，再按真实 outcome 重放。gem5 的恢复路径在 [decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/decoupled_bpred.cc#L977) 和 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L1227)。  
如果 RTL 恢复时没有把 IMLI counter 恢复到预测时快照，再用真实 `backward-taken` 重放，后面所有 `iIndex` 都会漂。

4. `commit/update` 窗口  
看 update 是否真的打到了预测时保存的 `pred.iIndex`，以及 update 条件有没有触发。gem5 只有“预测错”或“margin 不足”才训练，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L794)。

**3. 怎么区分是 IMLI 没学到、被 gate 挡掉，还是 alias / 更新时机问题**

可以按下面这套判别树来切。

**A. IMLI 没学到**

特征：

- `iIndex` 会随迭代变化，但 `iPercsum` 长期接近 0，或者符号不稳定。
- update 明明在打，但同一 phase 下的 `iTable` counter 不收敛。
- `useSc` 开不开都无所谓，因为 raw `iPercsum` 本身就不提供稳定方向。

在 gem5 里，这通常对应“表内容没形成有效偏置”，看 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L817)。  
离线分析时，可以直接看 `MGSCTRACE` 里的 `iPercsum` 符号是否经常和 `actualTaken` 对齐，见 [sc_first_principles.py](/nfs/home/yanyue/workspace/GEM5_review/util/sc_first_principles.py#L170)。

**B. 被 gate 挡掉**

特征：

- `iPercsum` 方向大多是对的，甚至 `totalSum` 方向也是对的。
- 但 `useSc=0` 很多，尤其集中在 TAGE high/mid confidence。
- 换句话说，SC“知道答案”，但没被允许覆盖。

gem5 的 gate 规则很直接：high/mid/low conf 分别要求 `abs(total_sum) > total_thres/2, /4, /8`，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L420)。  
仓库里的统计计数器已经能直接看这个现象：`scHighBypass/scMidBypass/scLowBypass`，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L731)。  
如果你在 gem5 上开 `--enable-bp-db=mgsc`，`MGSCTRACE` 会直接记录 `totalSum/totalThres/useSc`，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L226)。

**C. alias 问题**

这是当前实现里我最怀疑、也最值得对 RTL 对齐的点之一。

原因：

- 当前 MGSC 主表是“纯 counter 阵列”，没有 tag match；读取直接 `table[i][idx1][idx2]`，更新也直接回写这个位置，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L272) 和 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L579)。
- `idx1` 由 `startPC XOR folded_hist` 决定，`idx2` 只看 `branchPC` 低位，见 [btb_mgsc.hh](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.hh#L273)。

所以 alias 的典型症状是：

- 同一个 branch、同一个 phase，`iIndex` 没问题，但 counter 值被别的 branch 周期性污染。
- 你在 RTL 上会看到多个不同 `branchPC` 打到同一 `iTable` 行/槽。
- 训练了一阵后精度还是上下抖，不像“没学到”，更像“学了又被覆盖”。

离线时可以用仓库现成 SQL / 脚本看 branch-level 归因。`util/mgsc_table_probe.py` 会计算每个表对某个 branch 的 `decisive_ratio` 和 `agree_fix_ratio`，见 [mgsc_table_probe.py](/nfs/home/yanyue/workspace/GEM5_review/util/mgsc_table_probe.py#L264)。  
如果某 branch 的 `i_decisive_ratio` 很高但 `i_agree_fix_ratio` 不高，往往就是 alias 或错误 phase 输入，而不是单纯 gate。

**D. 更新时机 / 恢复时机问题**

特征：

- 预测时 `iIndex` 看起来合理。
- 但一旦发生 squash，后续几个动态实例的 `iIndex` 全部偏一拍或错 phase。
- 或者训练写入落在“不是本次预测看到的那个相位”。

这类问题通常不是“表学不会”，而是“spec state 和 recover state 不一致”。

要点：

- predict 侧先 spec update folded hist，见 [decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/decoupled_bpred.cc#L909)。
- squash 侧先 restore `target.history/phistory/bwhistory/lhistory`，再按真实结果 recover，见 [decoupled_bpred.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/decoupled_bpred.cc#L987)。
- IMLI recover 单独走 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L1253)。

如果 RTL 的波形里你看到：

- `pred_bw_taken` 和 `real_bw_taken` 定义不一致；
- 或 recover 没有恢复到 prediction snapshot；
- 或 commit/update 用的是 commit-time 现算 index，而不是 prediction-time snapshot；

那就应该归到“更新时机 / 恢复时机问题”，不是“IMLI 没学到”。

**4. gem5 当前 IMLI 可能还有哪些提升空间**

我认为至少有 6 个明确方向，而且都能从当前代码直接看出来。

- `weight scaling` 事实上没有启用。`calculateScaledPercsum()` 现在直接 `return percsum`，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L301)。但 weight 表还在训练，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L607)。这意味着当前 `iWeightTable` 对最终决策没有真实作用，属于最直接的提升点。

- `IMLI 语义过窄`。当前 IMLI 只看 backward-taken streak，而且来自 `getBwHistInfo()` 的“第一个 taken cond branch”，见 [common.hh](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/common.hh#L606)。如果 gobmk 的目标分支不是 fetch block 里最早的 taken cond，或者 loop 语义不是纯 backward-taken，gem5 现在的 IMLI 其实是在学一个替代信号，不是目标分支的真 iteration。

- `只有 1 个 I table`，默认 `iTableNum=1, iHistLen=[8], iTableIdxWidth=12`，见 [BranchPredictor.py](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/BranchPredictor.py#L1110)。对 gobmk 这类 phase 可能受外层上下文扰动的分支，单表通常不够，至少可以考虑多几何长度或 I+G/I+P 交叉。

- `没有 tag`，alias 风险很高。当前实现就是 tagless counter array，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L272)。如果要长期做 gobmk/spec06 热点优化，我会把“IMLI 表增加轻量 tag 或更强 hash”放到高优先级。

- `PC-indexed threshold 默认关闭`。`enablePCThreshold=False`，见 [BranchPredictor.py](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/BranchPredictor.py#L1154)。这意味着 gate 主要靠全局阈值；对少数特别适合 IMLI 的 branch，不够细。gobmk 这种 case 很适合试 PC-local threshold。

- `更新策略偏保守`。当前只有“错了”或“margin 小”才训练，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L794)。对稳定但稀疏的 loop-exit 相位，可能收敛偏慢。可以考虑对 `tage wrong, sc bypass` 的样本加速 I 表训练。

**建议你真正落地时怎么做**

先别直接盯 RTL 全波形，先把 gem5 侧证据收齐，然后拿同一组信号去对 RTL：

1. 开 `--enable-bp-db=mgsc`，抓 `MGSCTRACE`，因为它已经把 `iPercsum/totalSum/totalThres/useSc/actualTaken` 都记了，见 [btb_mgsc.cc](/nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L222)。

2. 用 [sc_first_principles.py](/nfs/home/yanyue/workspace/GEM5_review/util/sc_first_principles.py#L108) 先看这个 branch 的 `net_use`，再看 `i` 表的对齐度和净贡献。

3. 用 [mgsc_table_probe.py](/nfs/home/yanyue/workspace/GEM5_review/util/mgsc_table_probe.py#L264) 算 `i_decisive_ratio` 和 `i_agree_fix_ratio`。  
`i_decisive_ratio` 高但 `useSc` 少，多半是 gate。  
`i_decisive_ratio` 高但 `agree_fix_ratio` 低，多半是 alias / 错 phase。  
`i` 两个 ratio 都低，多半是根本没学到。

4. RTL 波形只抓我上面列的 shortlist。先对一次正确预测、一次错误预测、一次 squash-recover 后再预测，这三段通常就够分型。
