**例子 1：精确到 PC 和源码行的 helper 分支**

ELF 里我实际解到：

- `0x2f5656 -> __clzdi2 -> libgcc2.c:690`
- 同一小段里的分支有：
  - `0x2f5662: bnez a3, ...`
  - `0x2f5666: bnez a5, ...`

这段逻辑从汇编上看，本质是在做“按字节扫描，找第一个非零字节”，也就是 `count leading zeros` 之类的 helper。

直觉判断：

- 这种分支**不是 SPEC 算法主逻辑**
- 它更像底层位操作辅助函数
- 可预测性主要取决于输入值的位分布
- 如果输入高位非零的模式很稳定，它会比较好预测
- 如果输入值分布很散，尤其高位非零字节位置变化大，就没那么稳

这个例子对你的启发是：  
**看到 branch，不要先假设它一定对应 benchmark 里的 `if/for`。它可能只是编译器运行时 helper。**

**怎么把这个直觉迁移到你以后看的 branch**

你以后拿到一个 branch PC，可以先按下面顺序判断：

1. 它属于 benchmark 主体，还是 `libgcc/glibc/jemalloc/bbl`
2. 如果是 benchmark 主体，它是：
   - loop-exit
   - guard / fastpath
   - predicate result
   - regex / parser / symbol-state / pointer-state
3. 再判断它是不是依赖：
   - 简单长度/计数
   - 某个 load 出来的字段
   - 输入内容本身
   - 全局状态机/符号表/regex engine 状态

一个很粗糙但实用的经验是：

- `循环退出 / 范围检查 / 空指针检查`：通常更好预测
- `依赖输入内容真假 / regex 成败 / 查表状态 / load 值分类`：更可能 hard
