  最值得采纳的意见有这几条。

  1. UB-S(outcome) 和 “纯结构上界” 不能完全画等号
     这个我认同。
     因为如果只用 branchPC + exact outcome history 做 key，它已经顺手把一部分 path alias 也消掉了，所以它更像：

  - 一个很强的、偏乐观的 structural upper bound
  - 而不是“只去掉 set/way 冲突，其它都不变”的最小上界

  所以命名上最好更明确。
  我倾向于把第一版直接叫：

  - UB-S(outcome) 或
  - Per-Branch Exact-Outcome-History TAGE

  这样后面如果再做 UB-SP-lite(exact path)，语义会更清楚。

  2. 独立新类最好继承 BTBTAGE
     这个我也认同，而且这是个很关键的工程判断。
     review 说得对，当前 frontend 装配里本来就是围绕 BTBTAGE* 和 Param.BTBTAGE(...) 走的。
     如果完全独立不继承，改动面会不必要地放大。

  所以第一版最合理的形态应该是：

  - BTBTAGEUpperBound : public BTBTAGE

  复用这些现成能力：

  - history/spec update/recover 生命周期
  - TagePrediction
  - TageMeta
  - predictor integration path

  只替换：

  - provider/alt lookup
  - exact-key storage
  - allocation path

  这会比“全新平行实现一个 predictor”稳很多。

  3. updateOnRead 第一版应该关掉
     这个我也认同。
     对 current array-based TAGE，updateOnRead 的重查代价还行；
     但对 exact-history map 版，它会把最重的查找路径付两次。

  如果 upper-bound 的目标是“判别空间”，不是“完全复制当前微观行为”，那第一版更实用的做法是：

  - prediction 阶段把 provider/alt/key/table 存进 meta
  - update 阶段直接用 meta
  - v1 关闭 updateOnRead

  这是非常合理的 tradeoff。

  4. 内存和运行时间风险比我原文写得更重
     这个我也认同。
     尤其是后几张长历史表，如果 key 真做成 exact-history，unordered_map 的 node/bucket/pointer 开销会比 entry 本身还显眼。

  所以如果做实现，我会把下面这些当成 v1 必做项，而不是“优化项”：

  - 固定宽度 key，不用 dynamic_bitset 当 map key
  - 自定义 hash
  - 每表 reserve
  - 统计 contexts_per_table
  - 统计 contexts_per_hot_branch

  这一点 review 说得很对：

**背景**

这版 `UB TAGE` 的核心思路，和普通 `BTBTAGE` 不一样：

- 普通 `TAGE`：为了进定长表，必须把长历史 **fold** 成较短的 index/tag。
- 这版 `UB`：不想要 folding alias，所以直接拿“原始 history snapshot”做 exact key，放进 `unordered_map` 里查。[btb_tage_ub.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.cc#L112) [btb_tage_ub.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.cc#L522)

所以你看到的 `buildKey / words / activeWords / hash`，本质上都是在做一件事：

**把“branchPC + 这张表需要的那一段历史”变成一个可放进哈希表的 key。**

---

**1. 为什么是 `(histLen + 63) / 64`**

这是一个很常见的写法，意思是：

**计算“要存 `histLen` 个 bit，需要几个 64-bit word”**

也就是 `ceil(histLen / 64)`。

举例：

- `histLen = 4`，需要 `1` 个 `uint64_t`
- `histLen = 64`，需要 `1` 个 `uint64_t`
- `histLen = 65`，需要 `2` 个 `uint64_t`
- `histLen = 109`，需要 `2` 个 `uint64_t`
- `histLen = 397`，需要 `7` 个 `uint64_t`

如果你直接写 `histLen / 64`，那是向下取整：

- `65 / 64 = 1`
- 但 1 个 word 只能装 64 bit，不够装 65 bit

所以必须加 `63` 再整除，才能得到向上取整结果。[btb_tage_ub.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.cc#L130)

---

**2. 为什么不是“只要 64-bit 历史”**

因为当前这些表的历史长度本来就大于 64，比如默认是 `4, 9, 17, 29, 56, 109, 211, 397`。[BranchPredictor.py](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/BranchPredictor.py#L1055)

如果你只拿 64 bit：

- 那么 `109` 历史表、`211` 历史表、`397` 历史表
- 实际上都会退化成“只看最近 64 bit”

这样就不再是“exact long history”了，而是又引入了新的 alias。

所以这版 UB 才要支持多于 64 bit 的历史。

---

**3. `buildKey` 到底在干什么**

它不是在做什么复杂编码，本质很直接：

1. 先把外面传进来的完整 history bitset，按 64 bit 一块，拷到 `words[]` 里。[btb_tage_ub.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.cc#L112)
2. 对某一张表，只取这张表自己的 `histLen`
3. 于是 key 变成一个“元组”：

```text
key = (branchPC, low histLen bits of history)
```

代码里的 `ExactHistoryKey` 就是这个元组。[btb_tage_ub.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.hh#L37)

其中：

- `branchPC`：区分不同分支
- `words`：真正的历史内容
- `activeWords`：这次这张表到底用了几个 64-bit word

注意这里的 `pc + words + activeWords` **不是数学加法**，而是“这个 key 由这几个字段共同组成”。

---

**4. `activeWords` 是干嘛的**

因为 `words` 是个**固定长度数组**，比如最多 8 个 `uint64_t`。

但不是每张表都真的要用满 8 个 word：

- `histLen=17`，其实只用第 1 个 word
- `histLen=109`，只用前 2 个 word
- `histLen=397`，只用前 7 个 word

所以 `activeWords` 的作用就是告诉代码：

**这个 key 里，前多少个 word 才是有效历史。**

它主要有两个作用：

- 语义上明确“这张表实际用了多长历史”
- hash 时只混入有效 word，避免后面那些补零的尾巴干扰哈希分布

还有一件事你可能注意到了：

```cpp
const unsigned rem = histLen % 64;
if (key.activeWords > 0 && rem != 0) {
    key.words[key.activeWords - 1] &= ((1ULL << rem) - 1);
}
```

这一步是在把“最后一个 word 里多拷进去的无关 bit”清掉。

比如 `histLen=70`：

- 需要 2 个 word
- 但第 2 个 word 实际只该保留 6 bit
- 其余 58 bit 都要 mask 掉

这样才能保证 key 真正只代表“前 70 bit 历史”。

---

**5. 所以这版到底有没有用 folded history**

**没有。**

至少这段 key 生成逻辑里，没有用 folded history，而是直接用外部传进来的原始 history snapshot：

- `captureHistoryWords()` 是直接拷 bit block
- `buildKey()` 是直接按 `histLen` 截断
- `checkFoldedHist()` 在 UB 里是空实现

也就是说，这版 UB 的本意就是：

**不用 folding，直接 exact-match 历史上下文。**

---

**6. `ExactHistoryKeyHash` 是干嘛的**

这个纯粹是给 `std::unordered_map` 用的。[btb_tage_ub.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage_ub.cc#L29)

因为 `unordered_map` 需要两样东西：

- `operator==`：判断两个 key 是否真的相等
- `hash(key)`：先把 key 映射到哈希桶里

所以：

- `ExactHistoryKey` 定义“什么叫同一个上下文”
- `ExactHistoryKeyHash` 定义“怎么把这个上下文快速散列”

它**不改变语义**，只是为了让哈希表查得快。

你可以把它理解成：

- `operator==` 是最终裁判
- `hash` 是先粗分桶

即使 hash 冲突了也没关系，`unordered_map` 后面还会再用 `==` 做最终确认。

---

**7. 为什么不用 `dynamic_bitset` 直接当 key**

因为太重了。

这版实现故意把 key 做成：

- 固定大小 `std::array<uint64_t, N>`
- 再配一个 `activeWords`

这样比直接拿动态 bitset 当 map key 更轻，也更好自定义 hash。

所以这是一个很典型的工程折中：

- 语义上保留 exact history
- 实现上避免 dynamic container 当 key 太慢

---

**一句话总结**

`buildKey()` 做的事情其实很朴素：

**把“branchPC + 这张表要求长度的原始 history”整理成一个固定格式的 key。**

- `(histLen + 63) / 64`：算需要几个 64-bit word
- `words`：存这段历史内容
- `activeWords`：说明用了几个 word
- `ExactHistoryKeyHash`：让 `unordered_map` 能高效查这个 key

## Assistant (16:24)

**背景**

现在三条线可以这样理解：

- `UB-S(outcome)`：exact outcome history
- `UB-P(path-hash)`：exact 当前 `PHR`，但这个 `PHR` 本身已经是 lossy 的 `pathHash(pc, target)` 流
- `UB-SP-lite(raw path token)`：想测的是  
  **“如果不先做 15-bit hash + XOR，而是保留更原始一点的 path token，效果还能不能再上去”**

所以 `UB-SP-lite` 的核心，不是再做一个更大的 hash，而是：

**把每次 taken branch 的 path 信息，变成一个小而稳定的 token，按时间顺序保留下来，然后 exact-match 这个 token 序列。**

---

**我建议的 v1 定义**

我建议第一版做得很克制，定义成：

- 只对 `taken` branch 更新
- 每次更新生成一个 **固定宽度 token**
- token 宽度先用 **8 bit 或 10 bit**
- 保存最近若干个 token 的序列
- 每张表按“最近 N 个 token”做 exact key

也就是：

```text
key(table i) = (branchPC, last N_i path tokens)
```

这比当前 `UB-P` 强的地方在于：

- 不再把 `(pc, target)` 先压成 15-bit hash 再 XOR 到 bitset
- 保留了“token 的边界”和“时间顺序”
- 没有 XOR 混叠

但它又比“完整 raw path”轻很多，所以叫 `lite`。

---

**1. token 应该怎么定义**

我不建议一上来就把 `pc` 和 `target` 原样塞进去，那样上下文会碎得太厉害。

第一版更稳的 token 形式可以是：

```text
token = concat(
    branchPC[7:2]  的一部分,
    targetPC[7:2]  的一部分,
    branch type / call-return hint 的少量bit
)
```

更具体一点，我建议两个候选：

**方案 A：8-bit token**
- `pc_low6 ^ target_low6` 给 6 bit
- 再加 2 bit branch class
- 总共 8 bit

**方案 B：10-bit token**
- `pc_low5`
- `target_low5`
- 总共 10 bit
- 不做 XOR，保留 source/target 两侧一点独立信息

如果只想先探针，不想太碎，我更推荐：

- **v1 用 8 bit**
- **v2 再试 10 bit**

原因很简单：
- 8 bit 更像“时序友好的 token”
- 10 bit 更像“再往上探一点信息量”

---

**2. history 应该怎么存**

这里不要再用现在这种“bitset + XOR”的 `PHR` 语义了。

我建议单独维护一个 token history，例如：

```cpp
std::array<uint16_t, MaxPathTokens> exactPathTokens;
unsigned validTokens;
```

或者 ring buffer：

```cpp
std::array<uint16_t, MaxPathTokens> tokenRing;
unsigned head;
unsigned count;
```

我更推荐 **ring buffer**，因为语义更清楚：

- 每来一个 taken branch，就 push 一个 token
- recover 时直接恢复到 prediction-time snapshot，再 append resolved branch 的 token
- 不需要做整条 bitset 的大范围 shift

这对软件实现和 RTL 映射都更自然。

---

**3. 每张表到底用多少 history**

这是这个方案里最值得先定死的一点。

当前 `BTBTAGE` 的 `histLengths` 是 bit 长度：
`4, 9, 17, 29, 56, 109, 211, 397`

因为它会同时改两件事：

- 改了 path 表达方式
- 还改了有效历史深度

这会让实验结论变脏。

你刚才指出的核心问题很对：

- 如果 token 变成 `8 bit`
- 那同样 `397/512 bit` 预算下，能保留的 branch 个数会少很多
- 结果变差时，你根本分不清：
  - 是 token 表达不好
  - 还是只是历史长度不够了

所以在这个阶段做 `UB-SP-lite`，回答不了你最关心的问题。

---

**我建议先拆成这三个层次**

### 1. 先把 `PHR` 固定住
也就是当前这套：

- `taken` 才更新
- `(pc, target)` 先变成 `15-bit pathHash`
- 然后 XOR 进 `PHR` bitset

这套语义现在是确定的。[folded_hist.hh](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/folded_hist.hh#L22)

### 2. 先看 base TAGE 和 `UB-P` 差多少
这一步回答：

**“同样的 `PHR`，当前结构损失了多少信息？”**

这也是你现在最值得优先做的。

### 3. 等结构差距摸清后，再探索 `PHR` 编码
只有这时做 `UB-SP-lite` 或别的 `PHR` 增强方案，结论才干净。

---

**当前 base TAGE 和 `UB-P` 的差异，到底有哪些**

这里最好把“差异”拆成三类。

### A. 结构性差异
这是最大的差异，也是 `UB-P` 主要去掉的部分：

- base TAGE 依赖 folded history 算 index/tag
- 有 tag alias
- 有 set/way 容量限制
- 有 replacement / victimization
- 有 allocation 失败
- 同一个 exact path context 可能被别的 context 顶掉

而 `UB-P` 是：

- `key = (branchPC, exact current path-hash history)`
- `unordered_map` 精确命中
- 没有 set/way 冲突
- 没有 replacement 顶掉

这部分差异，正是你应该优先量化的。

### B. 训练/微观语义差异
这个差异比结构小，但也要注意：

- `UB-P` 现在是 `updateOnRead = false`
- base `BTBTAGE` 的银行仿真更偏向 `updateOnRead = true`，代码里也有 warning。[btb_tage.cc](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/pred/btb/btb_tage.cc#L104)

所以如果你要做严谨对比，最好把这类微观语义也尽量对齐或单独 A/B。

### C. 上游 history 编码差异
这部分 **base 和 `UB-P` 现在是相同的**

也就是：
- 两边都建立在同一个 `PHR` 语义上
- `UB-P` 没有提前换 raw path

所以 `UB-P` 是一个很好的“当前 `PHR` ceiling probe”。

---

**那接下来最合理的路线是什么**

我建议按这个顺序来。

## 第一阶段：把 `UB-P` 当作当前 `PHR` 的结构上界探针

目标不是再改 predictor，而是先回答：

**“base 和 `UB-P` 的 gap 主要来自哪里？”**

这一阶段最该补的不是新 predictor，而是**归因统计**。

我建议重点看这几类统计：

- base 中 provider miss / alt miss
- tag match miss
- allocation failure
- useful bit 阻塞分配
- long-history table 的命中率和替换率
- top mispredict branches 在 `base -> UB-P` 上的改善来源

一句话说：
**先知道 base 到底输在“找不到”、“撞错标签”、“分不进去”，还是“分进去了也学不好”。**

---

## 第二阶段：在不改 `PHR` 的前提下，把 base 往 `UB-P` 靠

这是你说的“先把当前可落地版本和 `UB-P` 差异尽可能弥补”。

我同意，而且这一步最有工程价值。

我建议优先看这几个方向：

### 1. 提高长历史表的结构承载能力
也就是优先减少高表 alias / replacement：

- 长历史表加 way
- 长历史表加 tag bits
- 长历史表增 entry
- 只对高表做这些，不要全表一起涨

因为通常 gap 最容易出在高表。

### 2. 改善 folding / index / tag 生成
保持同一 `PHR`，但让它被映射到表里的方式更少冲突：

- index/tag 不要过度共用同一 folded source
- 长表用更强一点的 folded 组合
- 尽量减少“不同 exact PHR 上下文映到同一 tag/index”的情况

这是最接近“在不改 history 的前提下榨干 `PHR`”。

### 3. 改 allocation / replacement policy
很多时候不是 history 不行，而是：

- 高表 entry 不够稳
- 分配太激进或太保守
- useful 机制把该学的上下文拦掉了

这里是很可能接近 `UB-P` 收益的地方。

### 4. 单独对齐 `updateOnRead` 影响
这个可以作为独立 A/B，不一定要改默认值，但要看清楚它对 gap 的贡献。

---

## 第三阶段：如果 base 仍明显落后 `UB-P`，再讨论 `PHR` 本身的增强

只有到了这一步，才应该问：

**“当前 `PHR` 编码是不是已经成为瓶颈？”**

如果这时 base 经过结构优化后仍明显落后 `UB-P`，那说明问题更多在：

- folding/tag/容量
- 而不是 `PHR` 本身

反过来，如果 base 已经很接近 `UB-P`，但整体效果还是不够好，那才说明：

- 当前 `PHR` 编码本身可能是上限

这时再做 `UB-SP-lite` 才有意义。

---

**如果想先探索 `PHR` 本身，但又不想引入“历史分支数变少”的混淆，怎么办**

这个点很关键。

如果你现在就想探 `PHR` 上限，我反而不建议先做 `raw path token`，而建议做 **`UB-P+` 类实验**，也就是：

**保持“历史时间跨度”大致不变，只增强 `PHR` 的信息量。**

比如：

### 方案 A：双 path-hash history
不是只有一条 `PHR`，而是两条独立 path-hash 流：

- 一条偏 `branchPC`
- 一条偏 `targetPC`
- 或两条不同 seed / 不同抽位方式的 hash history

然后 `UB` exact-match 这两条 history 的组合。

这样做的好处是：

- 仍然是固定宽度 bit history
- 仍然像 RTL-friendly 结构
- 不会像 `8-bit token` 那样一下子把可覆盖分支数砍掉很多

### 方案 B：更干净的 pathHash 编码
还是 1 次 event 推进少量 bit，但别再用单一 XOR 混入方式，改成：

- 多 lane path history
- lane 间独立更新
- predictor 侧联合使用

这个方向比 raw token 更贴近“RTL 可落地的 PHR 增强”。

---

**所以我对当前路线的建议是**

### 现在最值得做的
不是立刻 `UB-SP-lite`，而是：

1. 先把 `UB-P` 当 probe 跑起来
2. 分析 base 到 `UB-P` 的 gap 来源
3. 先优化可落地 TAGE 的结构部分
4. 看能不能把 base 向 `UB-P` 拉近

### 只有在这之后才该做
- `UB-SP-lite(raw path token)`
- 或其它更原始 path 表达

因为那一步是在探索“当前 `PHR` 之外还有没有额外信息”，而不是在解决当前 base TAGE 没吃干净 `PHR` 的问题。

---

**一句话总结**

是，**先弥补“当前可落地 TAGE”和 `UB-P` 的差异，再探索 `PHR` 本身**，这个顺序更对。

因为：

- `UB-P` 已经固定了 `PHR` 语义
- 它能先回答“结构浪费了多少”
- `UB-SP-lite` 现在做会把“表达更强”和“历史变短”混在一起，结论不干净

如果你愿意，我下一步可以直接帮你列一个“base 往 `UB-P` 靠”的具体优化清单，按：
- 最可能有收益
- 最接近 RTL 可落地
- 实现侵入性最小

这三个维度排个优先级。
