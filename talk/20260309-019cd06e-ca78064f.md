## Assistant (10:40)

**结论**

是的，**当前阶段不建议把 `--raw-cpt` 作为主启动路径**。如果你的目标是“先让 gem5 里的 Linux 跑起来”，更合适的是先走 **NEMU 生成 checkpoint，然后 gem5 恢复 checkpoint** 的路径，也就是：

```bash
export GCB_RESTORER=<gcpt_restorer>
./build/RISCV/gem5.opt ./configs/example/kmhv3.py --generic-rv-cpt /path/to/checkpoint.gz
```

这里**不要加** `--raw-cpt`。

**原因**

- 仓库文档自己就写了，`--raw-cpt` 理论上能跑，但“带 NEMU 后已经很久没有完整用 XS-GEM5 启动 Linux 了，初始化寄存器和 CSR 不能保证还能用，不建议直接启动”。见 [NEMU_XS-GEM5_usage.md](/nfs/home/yanyue/workspace/GEM5_review/docs/Gem5_Docs/top/NEMU_XS-GEM5_usage.md#L430)。
- `kmhv3.py` 这条路径在 `raw-cpt` 下走的是 `RiscvBareMetal` 语义，不是标准 `RiscvLinux` 启动链，当前配置里 `dtb_filename` 还是空的。见 [xiangshan.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/xiangshan.py#L215) 和 [config.ini](/nfs/home/yanyue/workspace/GEM5_review/debug/coremark/config.ini#L5208)。
- `kmhv3.py` 还会**默认强制开启 difftest**，除非 trace mode。见 [xiangshan.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/xiangshan.py#L717)。
- 我刚复现到的不是“纯 Linux 卡死”，而是 **OpenSBI 早期就在 `csrrs a0, time, zero` 处和 NEMU 分叉**，`mstatus/mepc` 已经不一致了。也就是说你现在被 **raw boot + CSR 语义 + difftest** 三件事一起干扰了。

**怎么选路径**

- 如果目标是“先证明 gem5 能跑 Linux 内核/用户态”：
  用 **NEMU checkpoint -> gem5 restore**。这是最稳的。
- 如果目标是“我要攻克完整上电启动链，从 OpenSBI 一路到 Linux”：
  可以继续研究 `linux.bin`，但**不要再用当前这条 `kmhv3.py --raw-cpt` 命令当第一落点**。它需要单独补齐：
  - 可关闭 difftest
  - 明确 DTB/bootargs 传递
  - 再看 OpenSBI/CSR/异常返回差异

**风险**

- 改走 checkpoint 路径，能证明“Linux in gem5 可运行”，但**不能证明 raw boot 已经对了**。
- 但这正是它的价值：如果 checkpoint 能跑，raw-cpt 不能跑，问题就被隔离到 **firmware/DTB/CSR/early boot**，范围会小很多。

**建议的下一步**

我建议先做两件事：

1. 先找或生成一个 **NEMU checkpoint**，用 `--generic-rv-cpt <checkpoint>` 不带 `--raw-cpt` 跑起来。  
2. 我继续帮你改 `kmhv3/xiangshan.py`，加上：
   - 显式关闭 difftest 的开关
   - 可选 `dtb_filename` / `command_line`
   - 这样再单独跑 `linux.bin`，把 raw boot 问题和 difftest 问题拆开

## Assistant (11:18)

**背景**

这几个概念很容易混在一起，因为它们都表现成“给 gem5 一个文件，然后跑起来”。但本质上它们不是一回事。

你现在接触到的至少有 4 类启动/恢复方式：

- `bare-metal` 启动
- `raw-cpt` 启动
- 标准 `Linux FS` 启动
- `checkpoint / 切片` 恢复启动

它们的核心差别在于：

- **guest 软件觉得自己是怎么开始执行的**
- **gem5 需要帮 guest 准备多少启动环境**
- **是否需要平台描述（DTB）**
- **是否是“从头启动”，还是“从中间状态恢复”**

---

**1. Bare-Metal 是什么**

`bare-metal` 最直观：

- 没有完整操作系统启动协议
- 没有复杂设备发现
- 通常就是“CPU 复位到某个地址，然后开始执行”

对 bare-metal 程序来说，常见假设是：

- 我知道入口地址
- 我知道内存布局
- 我甚至知道外设寄存器地址
- 不需要内核来枚举设备

所以 bare-metal workload 只需要很少东西：

- 把程序放到内存
- 设置 reset PC
- 开始跑

这就是为什么 `RiscvBareMetal` 路径原本很简单。

---

**2. raw-cpt 是什么**

你这里的 `--raw-cpt`，本质上不是“标准 Linux 启动模式”，而是：

- **把一个原始二进制镜像直接映射/拷入内存**
- 不经过普通 ELF/内核装载流程
- 然后从固定 reset vector 开始执行

所以它更像：

> “把一整个 payload 生硬地塞进机器内存里，然后假装这台机器上电了。”

如果这个 payload 恰好是：

- OpenSBI + Linux payload
- 或者某种 gcpt 拼出来的启动镜像

那么它“可能”能像真正启动一样工作。  
但前提是：**guest 期待的启动契约你得补齐。**

这就是这次的关键：

- 你塞进去的是 `linux.bin`
- 它里面有 OpenSBI / Linux
- 但 guest 仍然期待：
  - DTB 在哪里
  - `a1` 指向哪里
  - bootargs 是什么
  - 当前机器长什么样

`raw-cpt` 只是“把程序放进内存”，**不自动保证这些契约成立**。

---

**3. 标准 Linux FS 启动是什么**

标准 Linux FS 启动不是“单纯给一个 bin”。

它的语义更完整：

- 这是一台什么机器
- 内存有多大
- UART 在哪
- PLIC/CLINT 在哪
- 磁盘设备是什么
- rootfs 在哪
- bootargs 是什么

Linux 内核不是靠“猜”硬件运行的。  
尤其 RISC-V 上，Linux 很依赖 **设备树（DTB）** 来知道：

- CPU 节点
- memory 节点
- interrupt controller
- uart
- timer
- block device
- chosen/bootargs/stdout-path

所以标准 Linux FS 路径通常是：

1. 准备平台
2. 准备 DTB
3. 准备 kernel / bootloader / disk
4. 按 Linux 启动约定把参数传进去
5. 再开始执行

这和 bare-metal 最大区别就是：

- bare-metal 是“直接跑程序”
- Linux FS 是“按操作系统启动协议启动一台机器”

---

**4. checkpoint / 切片恢复是什么**

checkpoint / 切片恢复和前面两种的本质区别最大。

它不是“从头启动”。

它是：

> “CPU、寄存器、内存、CSR、页表、进程状态，都已经在某个时刻准备好了，我直接从那个中间状态继续执行。”

所以 checkpoint 恢复通常不需要再完整走：

- OpenSBI 初始化
- Linux early boot
- 设备发现
- init 进程拉起

这些都已经发生过了。

这就是为什么 checkpoint 路线特别适合：

- 性能实验
- 切片模拟
- 快速复现某个程序区间

本质上它是在跳过大量启动成本。

---

**5. 为什么 Linux 需要 DTB，而 raw bare-metal 往往不需要**

这是理解里最关键的一点。

### bare-metal 为什么常常不需要 DTB

因为 bare-metal 程序往往“写死了”平台知识。比如：

- UART 就在 `0x40600000`
- 内存从 `0x80000000` 开始
- 只有一个核
- 中断控制器我自己知道在哪

程序作者已经把“硬件描述”编进程序逻辑里了。

也就是说：

- **平台信息在代码里**
- 不需要运行时再给一份“机器说明书”

### Linux 为什么需要 DTB

Linux 内核是通用内核，它不应该为每一台具体机器重新硬编码一份地址表。

所以它更喜欢一种外部描述：

- “这是哪台机器”
- “这台机器有哪些设备”
- “这些设备地址、中断号是什么”
- “根设备和控制台是什么”

这份描述就是 **DTB**。

你可以把 DTB 理解成：

> “内核启动时收到的一份机器说明书”

没有这份说明书，Linux 就不知道：

- 串口在哪里，打印不到控制台
- 中断控制器怎么初始化
- 内存范围有多大
- timer/clint/plic 怎么接
- rootfs 从哪里来

所以 Linux 启动时，**DTB 不是可有可无，而是平台契约的一部分**。

---

**6. 为什么这次要描述“香山机器”**

因为 Linux 不关心“你说你这是 gem5”，它关心的是：

- 这台 RISC-V 机器有哪些硬件资源
- 它们映射到什么地址
- 用什么兼容字符串驱动匹配

你这次不是在启动一个抽象 Linux，而是在启动：

- 一个 **Xiangshan 风格内存布局**
- 一个 **Xiangshan 风格 UART/CLINT/PLIC/MMC 地址空间**
- 一个 **gem5 模拟出来的具体机器**

所以 DTB 必须描述“这台机器”，而不是泛泛描述“某个 RISC-V”。

如果 DTB 说的是 HiFive，
但你实际机器地址是 Xiangshan/NEMU 风格，那么 Linux 读出来就会错：

- 串口地址错
- 中断控制器错
- timer 错
- 设备发现错

这也是为什么你这次一旦把 **Xiangshan 机器的 DTB** 接上，事情突然就通了。

---

**7. raw-cpt 为什么之前也“像 Linux”，但其实职责不清**

因为 `linux.bin` 里包含的是 Linux 相关 payload，所以看起来像是在做 Linux 启动。

但从软件架构上：

- `raw-cpt` 的职责只是“把镜像放进内存并起跑”
- Linux 启动要求的是“给我一台描述完整的机器”

所以当你把 Linux 跑在 raw 路径上时，本质是在做一件职责越界的事：

- bare-metal/workload 层开始承担 Linux 启动契约

这就是为什么你前面觉得它“不清晰”，这个感觉是完全对的。

---

**8. 切片启动为什么通常不需要你再纠结 DTB**

因为切片/checkpoint 恢复时，Linux 通常已经活过了 early boot。

也就是说：

- DTB 已经被 bootloader/OpenSBI/内核消费过
- 页表已经建好了
- 驱动已经初始化了
- 运行上下文已经建立了

**3. RISC-V 启动时 `a0/a1` 的约定**

这是最核心的知识点。

RISC-V Linux 启动时，通常约定：

- `a0`：hart ID
- `a1`：DTB 地址

你可以把它理解成：

- `a0` 告诉内核“我是哪个核启动的”
- `a1` 告诉内核“设备树说明书放在内存哪里”

所以当 Linux 或上一级固件进入下一阶段时，它会非常依赖这两个寄存器。

- 它不是“小参数没传到”

---

**4. DTB 为什么不是“Linux 自己从磁盘读”**

因为 Linux 在 early boot 那个时刻：

- 还没完成设备初始化
- 还没挂载文件系统
- 还不知道 block device
- 甚至串口都可能还没 fully online

所以它不可能靠“正常文件 IO”去读一份设备树。

DTB 必须在它启动之前，就已经：

- 放在内存里
- 地址有效
- 通过启动协议告诉它在哪

所以启动链要做的是：

1. gem5 把 DTB 放进 guest 物理内存
2. 把 DTB 地址写进 `a1`
3. OpenSBI / Linux 按约定读取 `a1`
4. 解析 DTB
5. 再开始初始化平台

这正是你这次打通的关键。

**5. OpenSBI 在这里扮演什么角色**

- 提供 SBI 调用接口
- 管理一些 machine-mode 事务
- 把控制权转给 supervisor-mode 的 OS
- 在某些启动方式下，把 hartid / dtb / next stage 信息交给 Linux

所以你看到 `Boot HART ...` 那些信息，其实说明：

- 机器已经成功执行到 OpenSBI 阶段
- CPU、CSR、M-mode 至少已经工作了一部分
- 问题往往发生在“OpenSBI 往 Linux 交接”的环节

**8. DTB 里面到底有什么**

- `/cpus`

  - hart ID

  - plic

  - 中断号

- 定时器不工作
- 中断不工作

2. Boot ROM / OpenSBI / Bootloader 先运行
3. 它把控制权交给 Linux kernel
4. Linux 读取 DTB，初始化设备和内存
5. Linux 继续启动到用户态

所以输入文件通常会落在这几类角色里：

- **内核本体**
- **固件 / SBI / bootloader**
- **固件 + 内核的打包结果**
- **根文件系统 / 磁盘镜像**
- **设备树 DTB**

---

**2. `vmlinux` 是什么**

`vmlinux` 通常是：

- **未压缩、带符号的 ELF 内核文件**

特点：

- 是 ELF
- 往往有符号表，适合调试
- 更像“开发/调试产物”
- 不一定是最终给 bootloader 直接喂的格式

你可以把它理解成：

> “最完整的内核可执行文件，适合调试器和开发者看。”

它常用于：

- `objdump`
- `gdb`
- 符号定位
- 分析内核函数地址

但有时 bootloader/模拟器未必直接吃它。

---

**3. `Image` 是什么**

`Image` 通常是：

- **Linux 内核的裸二进制映像**
- 常见于 ARM / RISC-V 等架构

特点：

- 不是 ELF
- 通常不带调试符号
- 更接近“给 bootloader 加载的最终 kernel image”

可以理解成：

> “适合被启动器直接搬到内存再跳转执行的内核镜像。”

所以很多标准 FS 启动脚本更喜欢 `Image` 而不是 `vmlinux`。

---

**4. `fw_jump.elf` 是什么**

这是 OpenSBI 的一种固件产物。

`fw_jump.elf` 的意思大概是：

- **OpenSBI 固件本体**
- 它启动后会“跳转”到一个外部给定的下一阶段地址
- 下一阶段通常是 Linux kernel

也就是说这时输入是分开的：

- 一个 `fw_jump.elf`
- 一个 `Image` / `vmlinux`
- 一个 DTB

启动关系像这样：

- gem5/QEMU 先执行 `fw_jump.elf`
- OpenSBI 初始化 machine mode / SBI
- 然后跳到 kernel
- 再把 DTB/hartid 传给内核

所以 `fw_jump.elf` 适合“**分离式启动**”。

---

**5. `fw_payload.elf` 是什么**

这也是 OpenSBI 产物，但和 `fw_jump` 不同。

`fw_payload.elf` 的意思大概是：

- **OpenSBI + 下一级 payload 打包在一起**
- 这个 payload 常常就是 Linux kernel

也就是说：

- 固件和内核已经被预先打包成一个 ELF

启动关系像这样：

- gem5/QEMU 只需要加载一个 `fw_payload.elf`
- OpenSBI 启动后直接把控制权交给它内部携带的 payload

它比 `fw_jump + Image` 更“一体化”。

你可以把二者对比成：

- `fw_jump.elf`：固件和内核分开发
- `fw_payload.elf`：固件里已经夹带了内核

---

**6. `bbl` 是什么**

`bbl` 是更早期 Berkeley Boot Loader 的老方案。

历史上很多 RISC-V Linux 启动会用：

- `bbl`
- 或 “带 Linux payload 的 bbl”

它做的事情和 OpenSBI 某些打包方式有相似点：

- 负责最早期启动
- 再把控制权交给 Linux

现代 RISC-V 更常见的是 OpenSBI 体系，所以现在通常更关心：

- `fw_jump.elf`
- `fw_payload.elf`

而不是 `bbl`。
