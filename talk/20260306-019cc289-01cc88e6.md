**1. 串口输出路径是否可能在 OpenSBI 后失效**

结论：**可能，而且概率不低。**

证据：

- baremetal Xiangshan FS 平台挂的是 `UartLite + Clint + NemuMMC + NemuPlic`，不是完整的 Linux FS 平台。见 [FSConfig.py](/nfs/home/yanyue/workspace/GEM5_review/configs/common/FSConfig.py#L660)。
- 其中串口是 `UartLite`，地址是 `0x40600000`。见 [Uart.py](/nfs/home/yanyue/workspace/GEM5_review/src/dev/serial/Uart.py#L85)。
- `UartLite::write()` 只有往 `TX_FIFO` 写 1 字节时才直接 `putc(..., stdout)`；如果软件继续沿用这条 MMIO TX 路径，输出会直接到终端。见 [uartlite.cc](/nfs/home/yanyue/workspace/GEM5_review/src/dev/serial/uartlite.cc#L24)。
- 但 `UartLite::read()` 只实现了 `STAT_REG`，而且固定返回 `0`；其他寄存器访问直接 `warn`。见 [uartlite.cc](/nfs/home/yanyue/workspace/GEM5_review/src/dev/serial/uartlite.cc#L6)。
- 更关键的是，`RiscvUart8250` 有 `generateDeviceTree()`，`UartLite` 没有；也就是说这个 `UartLite` 没有现成的 DTB 描述生成逻辑。见 [Uart.py](/nfs/home/yanyue/workspace/GEM5_review/src/dev/serial/Uart.py#L70) 和 [Uart.py](/nfs/home/yanyue/workspace/GEM5_review/src/dev/serial/Uart.py#L85)。

这意味着：

- OpenSBI 早期 console 可能是“硬编码地址 + 轮询发字节”，所以还能打印。
- Linux 后续如果改走“按 DTB 识别串口驱动”或依赖更完整的 UART 寄存器/中断语义，就可能直接静默。
- 这类静默不等于没前进，只是“后续 console handoff 失败”。

**2. `NemuPlic` 是否足以导致 Linux 后续卡住或静默**

证据非常直接：

- Linux 如果探测/配置 PLIC，读到的始终是 0，外部中断路径基本不可用。
