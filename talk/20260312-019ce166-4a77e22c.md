**结论**

Ubuntu 编译坏掉的点我复现出来了，而且已经顺手修掉并验证过。

原始失败发生在构建 `build/RISCV/mem/mem_util.o` 时，报错是：

```text
/usr/include/boost/config/stdlib/libstdcpp3.hpp:267:11:
error: missing binary operator before token "("
#     if !_GLIBCXX_DEPRECATED
```

**根因**

问题来自 [`src/mem/mem_util.cc:15`](/nfs/home/yanyue/workspace/GEM5_review/src/mem/mem_util.cc#L15) 这段为了 macOS 加的 Boost workaround。

你这次提交里把下面两个宏无条件定义了：

- `BOOST_LIBSTDCXX_VERSION`
- `BOOST_LIBSTDCXX_VERSION_WORKAROUND_GUARD`

这在 macOS + `libc++` 上是合理的，但在 Ubuntu + GCC + `libstdc++` 上会污染 Boost 对标准库版本的正常检测，导致它把 `BOOST_LIBSTDCXX_VERSION` 误认为 `0`，然后错误走进旧版 `libstdc++` 分支，最后在 `_GLIBCXX_DEPRECATED` 这个 GCC attribute 宏上炸预处理。

**我做的修复**

我把这段 fallback 收紧到了真正需要的场景：

- 只在 `__APPLE__ && _LIBCPP_VERSION` 时启用

修改位置：
[`src/mem/mem_util.cc:15`](/nfs/home/yanyue/workspace/GEM5_review/src/mem/mem_util.cc#L15)

核心变化就是把原来无条件生效的 workaround 改成：

```cpp
#if defined(__APPLE__) && defined(_LIBCPP_VERSION)
...
#endif
```
