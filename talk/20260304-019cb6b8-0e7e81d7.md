- `p_update_thres = 0`

也就是说，**SPEC06 这里几乎只受全局 `updateThreshold` 控制**，PC 级 threshold 表没有参与。

---

**`total_thres` 的参与点**

在 [`btb_mgsc.cc:413`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L413 ) 到 [`btb_mgsc.cc:437`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L437 )：

1. `total_thres` 先由  
   `total_thres = (updateThreshold / 8) + p_update_thres`
   计算出来。
2. 它直接参与 **是否让 SC 覆盖 TAGE** 的 chooser gate：
   - 高置信 TAGE：`abs(total_sum) > total_thres / 2`
   - 中置信 TAGE：`abs(total_sum) > total_thres / 4`
   - 低置信 TAGE：`abs(total_sum) > total_thres / 8`

所以 `total_thres` 是一个**使用门槛**，控制“SC 证据有多强时才允许 override TAGE”。

**2. clamp 到非负后，哪些原本会发生的训练/override 被抑制**

**核心异常状态：旧代码允许 `updateThreshold < 0`**

在 `2ca965681b` 中，如果长期发生“预测正确但 margin 小”的更新，`updateThreshold` 会不断减小，最终可以变成负数。

一旦 `enablePCThreshold = False`，就有：

- `total_thres = updateThreshold / 8`

于是 `total_thres` 也可能变成负数。

---

**负 threshold 时，override gate 会退化**

旧逻辑里：

- 高置信：`abs(total_sum) > total_thres / 2`
- 中置信：`abs(total_sum) > total_thres / 4`
- 低置信：`abs(total_sum) > total_thres / 8`

如果 `total_thres < 0`，右边就是负数。于是只要 `abs(total_sum) >= 0`，条件几乎恒成立。

这意味着原本不会发生的行为会发生：

- **SC 会在几乎所有非零 `total_sum` 上覆盖 TAGE**
- 即使 `total_sum` 很弱，也会 override
- 本来 threshold 应该表达“SC 证据是否足够强”，这里却变成了“几乎永远足够强”

`2a454fccdb` 在 [`btb_mgsc.cc:416`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L416 ) 到 [`btb_mgsc.cc:418`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L418 ) 把它 clamp 到 `>= 0`，抑制的正是这类**弱证据 override**。

---

**训练侧的抑制更微妙**

`2a454fccdb` 还在 [`btb_mgsc.cc:661`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L661 ) 到 [`btb_mgsc.cc:665`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/btb/btb_mgsc.cc#L665 ) 把全局 `updateThreshold` 本身钉在 `>= 0`。

这会抑制旧代码中原本会发生的一整段“演化路径”：

1. 正确但低 margin 时，进入 update path
2. `updateThreshold` 被持续 decrement
3. 降到负数
4. 下一轮预测几乎总是启用 SC override
5. MGSC 进入一种“异常激进”的工作状态

clamp 后，这条路径被截断在 `0`。

---

**但要注意：在 `2a454fccdb` 这个点上，训练门控本身仍然偏紧**

因为那时更新条件还是旧的：

- `wrong || abs(sum) < total_thres`

所以当 `total_thres == 0` 时：

- 正确预测的低 margin 更新条件变成 `abs(sum) < 0`
- 永远为假

结果是：

- threshold 不再进入负数区
- 但一旦掉到 0，正确但弱置信的样本也不再训练
- 这会让学习变保守

这正好解释了为什么后续 `7bfd326f15` 又把它改成 `abs(sum) < total_thres / 2`：作者显然也意识到**只修正“非负”还不够，更新窗口还要按 RTL 再放回去**。

**3. 为什么可能在某些 workload 上掉一点分，但更接近 RTL**

**先说为什么会掉一点分**

从纯代码逻辑看，`2ca965681b` 的负 threshold 状态会带来一种 simulator-only 的“超激进 SC”：

- TAGE 即使高置信，只要 SC 的 `total_sum` 非零，几乎就能抢过来
- 而当前配置只开了 `I + P + Bias` 三类表
- 这三类表对某些控制流模式很容易形成“弱但方向正确”的 signed sum

这会带来一个现象：

- 某些 branch 上，TAGE 虽然自信，但就是错
- MGSC 的 IMLI/path/bias 只给出很小的、但方向碰巧正确的 `total_sum`
- 在 bug 状态下，这点弱证据也足以 override
- 于是 benchmark 分数可能多出一点

这类收益并不一定稳定，但它确实可能存在。

---

**再说为什么更接近 RTL**

`threshold` 从命名、注释和同类实现语义上都更像是**非负置信门槛**，不是 signed bias。

可对照 gem5 里其它 SC 代码：

- [`statistical_corrector.cc:349`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/statistical_corrector.cc#L349 ) 到 [`statistical_corrector.cc:352`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/statistical_corrector.cc#L352 )
- [`multiperspective_perceptron_tage.cc:573`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/multiperspective_perceptron_tage.cc#L573 ) 到 [`multiperspective_perceptron_tage.cc:579`]( /nfs/home/yanyue/workspace/GEM5_review/src/cpu/pred/multiperspective_perceptron_tage.cc#L579 )

尤其 MPP 版本甚至对 `pUpdateThreshold` 负值直接拉回 0。

所以 `2a454fccdb` 的意义是：

- 去掉“负 threshold 导致门控失效”的软件伪影
- 让 threshold 真正只表达“证据强度门槛”
- 行为上更像硬件里常见的 unsigned / non-negative threshold 语义

分数略降，不代表修错了；更可能代表之前的实现**额外乐观**。

---

**4. 最值得怀疑的 benchmark 类型**

如果只按代码逻辑怀疑，我会优先盯下面这类：

**最可疑类型：整数类、控制流密集、路径相关强、带循环计数结构的 benchmark**

原因直接来自当前开启的 3 张表：

- `ITable`：擅长 loop trip count / modulo / nested-loop 节奏
- `PTable`：擅长 path-sensitive branch
- `BiasTable`：擅长强偏置分支

因此最可能受“负 threshold 异常 override”影响的是：

- 热点分支多、CFG 复杂
- 分支方向受前驱路径影响强
- 有明显 loop iteration / loop exit 结构
- TAGE 偶尔高置信错，但 IMLI/path 的弱信号常常方向是对的

对应到 SPEC06 类型，优先怀疑：

- 编译器/解释器/文本处理类
- 棋类/搜索类
- 图遍历/树遍历/规则系统类
- 含大量短循环、嵌套循环、phase 切换的整数 workload

如果要点名“高怀疑候选”，我会把这类放前面：

- `perlbench`
- `gobmk`
- `sjeng`
- `hmmer`
- `xalancbmk`

不是说一定是它们掉分，而是从 **IMLI + path + bias** 的受益画像看，它们最像会被“异常激进的弱证据 SC override”影响的对象。

相反，不太优先怀疑的是：

- 大量规则数值循环、控制流相对稳定、TAGE 已经足够好的 FP 类 benchmark

---

**一句话归纳**

`2ca965681b` 里，负的 `updateThreshold/total_thres` 会把 MGSC 的“置信门控”悄悄变成“几乎总是相信 SC”，这可能给少数 SPEC06 整数型 workload 带来一点 accidental boost；`2a454fccdb` 把这种异常状态拿掉后，分数可能略降，但语义上更合理，也更像 RTL。后续 `7bfd326f15` 再把 update 条件从 `< total_thres` 改成 `< total_thres / 2`，基本就是在补上“修正后训练窗口过紧”的第二步。
