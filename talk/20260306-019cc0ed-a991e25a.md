**核心结论**
这个 PR（主要是 `d966055b9d` + `fdba12d73e`）的核心不是“改算法”，而是“改流水线时序建模方式”：
1. 把 `tick` 顺序从前到后改成后到前（`commit -> iew -> rename -> decode -> fetch`）。
2. 把“阻塞传播”从 `timeBuffer` 里的 `block/unblock` 状态机，改成同周期共享的 `StallSignals`。
3. 目标是为后续 SMT 做“按级、按线程”更直接的背压传播，减少复杂状态机。

**3) 为什么会影响 stall/block/unblock/squash 传播**
这是这个 PR 的本质：

1. 前到后 tick 的问题  
- 后端（比如 commit/iew）发现需要阻塞时，前端（fetch/decode）本周期已经跑完，信号天然“晚一拍”  
- 所以之前要维护 `block/unblock` 状态机（`decodeBlock/decodeUnblock/...`）来补偿时序差

2. 后到前 tick 的收益  
- `commit` 先计算 `blockIEW`，`iew` 立刻用它算 `blockRename`，`rename` 再算 `blockDecode`，`decode` 再算 `blockFetch`，最后 `fetch` 当周期就能看到  
- 传播链变成“同周期组合式背压”，代码更直观  
- 典型链路可看：  
  - commit 写 `blockIEW`：[commit.cc#L1852](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/commit.cc#L1852)  
  - iew 读 `blockIEW` 写 `blockRename`：[iew.cc#L826](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/iew.cc#L826)  
  - rename 读 `blockRename` 写 `blockDecode`：[rename.cc#L344](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/rename.cc#L344)  
  - decode 读 `blockDecode` 写 `blockFetch`：[decode.cc#L441](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/decode.cc#L441)  
  - fetch 最后消费 `blockFetch`：[fetch.cc#L1292](/nfs/home/yanyue/workspace/GEM5_4/src/cpu/o3/fetch.cc#L1292)

3. squash 仍走 `timeBuffer`（不是 `StallSignals`）  
- squash 携带 PC、seq、mispredictInst 等复杂信息，仍通过 `commitInfo/decodeInfo/iewInfo` 传  
- `StallSignals` 只负责“快路径背压”，职责更单一
