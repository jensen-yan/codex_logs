# 我是如何把 Codex 用成分支预测研究助理的

这篇笔记不是总结某一条 prompt，而是总结我在一段时间内，如何把 Codex 逐步用成一个能协助做前端/BPU/TAGE/SC 研究的半自动研究助手。

核心结论很简单：

1. 先把高频、稳定、可复用的流程沉淀成 skills。
2. 再把一次复杂分析拆成“技能调用 + 自迭代实验 + 文档沉淀 + 异步 CI 回收”四段式流程。
3. 最后把 prompt 的目标从“让 AI 回答我”升级成“让 AI 帮我推进研究闭环”。

---

## 一、总的方法论

从这些会话里，我现在更认可一种工作流：

1. `skills` 负责固定套路。
2. `subagent` 负责并行搜集信息、跑 side task、减轻主上下文压力。
3. 主 agent 负责做决策、串联结果、落文档、改代码、跑实验。
4. `Markdown 文档` 不是附属品，而是研究过程中的中间产物。
5. `CI` 不是单纯拿分工具，而是异步实验平台；本地分析脚本则负责把 CI 结果重新转回“可解释证据”。

换句话说，我不是把 Codex 当问答机器人，而是当：

- 流程编排器
- 实验执行器
- 文档助手
- 数据分析助手
- 代码实现助手

---

## 二、模式 1：先把高频步骤沉淀成 skills

这是最重要的一条。如果没有这一步，后面所有复杂任务都会不断重复解释环境、脚本路径、输入输出格式，prompt 会越来越长，可靠性也会越来越差。

### 典型案例 A：把批量回归和分析脚本抽成 skill

代表会话：

- [prompts/20260227-019c9e72-bbf71e28.md](prompts/20260227-019c9e72-bbf71e28.md)

这组 prompt 很典型，先是让 Codex 修改已有脚本去调用 skills 分析批量结果，后面很快就收敛到一个更稳定的结论：

- 直接把 `run_cpt_back.py` 做成单独 skill
- 把脚本移动到 skills 目录的 `scripts/` 中
- 避免依赖某个 worktree 下的 `debug/` 脚本路径
- 输出最好变成 `csv/json`，方便后处理和批分析

这背后体现出的不是“想整理目录”，而是一个更强的意识：  
只要某个流程未来会重复调用，就不应该继续让它停留在一次性 prompt 里。

### 典型案例 B：把“分支可预测性归因”抽成 skill

代表会话：

- [prompts/20260304-019cb6ef-64ebfb7a.md](prompts/20260304-019cb6ef-64ebfb7a.md)
- [prompts/20260304-019cb793-76e915c6.md](prompts/20260304-019cb793-76e915c6.md)

这些 prompt 最初是围绕“某个 branch PC 能不能映射回源码、它到底是天然难预测还是算法没学好”展开的。很快，需求从一次性分析升级成一种稳定分析模式：

- 找 ELF / 函数 / 源码位置
- 判断 branch 更像 loop-exit、predicate、state-machine，还是 load-dependent
- 推断它更可能是语义难预测，还是预测器实现没学好

一旦这套分析链条稳定下来，最好的做法就不是继续手工 prompt，而是把它固化成一个 skill。

### 典型案例 C：把“CI 跑分回收 + 本地分析”抽成 skill

代表会话：

- [prompts/20260309-019cd1c9-b0d9bd93.md](prompts/20260309-019cd1c9-b0d9bd93.md)

这里已经非常接近成熟工作流了：

- 先让 Codex 去 `gh` 上看 commit 对应的 CI
- 再读取 summary / archive path
- 再调用本地 `gem5_data_proc`
- 最后还进一步要求把整套流程写成中文 skill，方便以后复用和分享

这条会话很能说明一个经验：  
真正有价值的不是“这次分析做完了”，而是“这次分析能否被下次直接重放”。

### 这一类 prompt 的共性

- 明确说出“这件事以后还会重复做”
- 主动要求沉淀为 skill，而不是脚本散落在 `debug/`
- 要求路径独立于某个特定 worktree
- 要求输出结构化结果，如 `csv/json/md`

## 三、模式 2：让 Codex 形成自迭代实验闭环

这是第二个关键能力。也就是不只是“帮我分析一下”，而是让它自动走完：

1. 跑切片
2. 分析数据
3. 改参数或改代码
4. 重新编译
5. 再跑切片
6. 再分析
7. 顺手把过程沉淀到 Markdown 文档里

### 典型案例 A：SC 开关实验 + 原理回推

代表会话：

- [prompts/20260303-019cb26f-e53bc68e.md](prompts/20260303-019cb26f-e53bc68e.md)

这条会话展示的是一个非常完整的实验闭环：

- 先调用 `$run-cpt-regression`
- 对 `kmhv3.py` 单独开关 SC
- 比较计数器差异
- 如果没效果，就继续问“是测试写得不好，还是 TAGE 已经预测得太好了”
- 接着继续往前走，要求从第一性原理分析哪些 branch 应该由 SC 纠正
- 再要求写新的小测试、编译运行、必要时打 trace

这类 prompt 的价值在于，它没有把 Codex 限制成“帮我做一件事”，而是允许它围绕目标不断收敛。

### 典型案例 B：从 CI 结果出发，回到本地做针对性实验

代表会话：

- [prompts/20260305-019cbbe7-3e4e872c.md](prompts/20260305-019cbbe7-3e4e872c.md)
- [prompts/20260310-019cd599-5cfc62f4.md](prompts/20260310-019cd599-5cfc62f4.md)
- [prompts/20260310-019cd749-cd7fd239.md](prompts/20260310-019cd749-cd7fd239.md)

这类会话已经是比较成熟的“研究代理”用法：

- 先读已有文档
- 再看最新 CI 结果和 archive path
- 再定位具体切片，如 `gobmk_nngs_18098`
- 再本地开 `--enable-bp-db tage` 跑短窗口
- 再分析 `topMispredictsByBranch.csv` / bp.db / branch-level evidence
- 然后再改参数或改代码验证猜想
- 最后把中间结论写回文档

这比传统“人工看表格 -> 人工猜原因 -> 再手工跑”要高效很多，因为 Codex 能在一次任务里把“证据链”串起来。

### 让自迭代闭环更稳定的 prompt 写法

从这些记录里，我现在更倾向于这么写：

- 给出明确目标，不只给一个命令
- 明确允许 `skills` 和 `subagent`
- 给出关键数据路径、checkpoint 路径、配置路径
- 允许它在需要时“改参数 / 编译 / 再跑 / 再分析”
- 明确要求把中间发现写入 `md` 文档
- 明确停止条件，例如“差不多可以了”“trace 已经太大了”

也就是说，prompt 要像给一个研究助理下任务，而不是像给 shell 下命令。

## 四、模式 3：文档先行，再进入实现

这是我觉得最值得推荐给同组同学的一种用法，尤其适合复杂设计题。

### 典型案例：upper-bound TAGE

代表会话：

- [prompts/20260310-019cd599-5cfc62f4.md](prompts/20260310-019cd599-5cfc62f4.md)
- [prompts/20260310-019cd6ce-0578a039.md](prompts/20260310-019cd6ce-0578a039.md)
- [prompts/20260310-019cd749-cd7fd239.md](prompts/20260310-019cd749-cd7fd239.md)

这几条会话非常清楚地展示了一个好流程：

1. 先从已有数据和已有文档出发，弄清楚问题到底是什么。
2. 不急着写代码，先讨论“为什么 8 ways 还有剩余 mispred”“能不能做 upper bound”。
3. 先产出一个设计文档。
4. 再让另一个 AI 或 reviewer review 设计文档。
5. 把 review 意见收敛成可执行工程结论。
6. 再决定由谁来实现。
7. 实现后先跑 `coremark` / smoke test / 单元测试。
8. 再回到更贵的切片实验和 CI。

这比“脑中大概有个想法，直接开写”稳太多，尤其对 upper-bound / oracle / idealized predictor 这种很容易语义漂移的实现尤其重要。

### 这类工作流为什么有效

因为它把复杂任务拆成了三个完全不同的阶段：

- `概念收敛`
- `工程边界收敛`
- `实现与验证`

如果一开始就直接让 AI 写代码，最容易出的问题不是编译不过，而是：

- 语义写偏了
- 上界定义不清
- 结果没法解释
- 后面实验结论不可信

而文档先行，实际上是在帮后续实验“保真”。

## 五、模式 4：参数修改 + push 触发 CI + 回收结果分析

这个模式是异步实验流里最实用的一种。

### 典型案例

- [prompts/20260309-019cd1e8-e78abb58.md](prompts/20260309-019cd1e8-e78abb58.md)
- [prompts/20260309-019cd1c9-b0d9bd93.md](prompts/20260309-019cd1c9-b0d9bd93.md)
- [prompts/20260310-019cd599-5cfc62f4.md](prompts/20260310-019cd599-5cfc62f4.md)

典型链路是：

1. 改一个小参数或者一组配置
2. `commit` / `push`
3. 触发远端 CI 跑 0.3c / 0.8c 数据
4. 一小时后回来
5. 让 Codex 去 `gh` 上拿 run、summary、archive path
6. 再用本地 `gem5_data_proc` 分析 benchmark 级变化、MPKI 变化和关键 counters
7. 如果这套流程后续要长期复用，再继续沉淀成 skill

这类 prompt 的价值很大，因为它把“CI 是一个黑盒”变成了“CI 是一个异步实验后端”。

### 这个模式里最重要的 prompt 要素

- 直接给出 `run url`、`archive path`、`score.txt path`
- 明确说出“先不要全部下载，summary 里先看什么”
- 明确要求本地脚本做二次分析，而不是只看总分
- 指出最关心的 benchmark / case，例如 `gobmk`、`sjeng`
- 如果还会重复用，顺手要求把流程写成 skill

## 六、这些案例背后的共通 prompt 风格

如果只总结成几条经验，我会写成下面这样。

### 1. 不要只提问，要给任务闭环

弱 prompt：

> 帮我分析下这个结果。

强 prompt：

> 帮我先看 CI 数据，再本地跑对应切片验证，再把中间结论写到文档里；如果需要可以改参数重编译再跑。

### 2. 不要只给主题，要给路径

这些高质量会话基本都有非常具体的路径：

- 文档路径
- checkpoint 路径
- CI archive path
- `score.txt` 路径
- `gem5_data_proc` 路径
- 具体配置和命令

路径一具体，Codex 就能直接进入执行态，而不是先花很多轮确认环境。

### 3. 不要只要求“分析”，要要求“沉淀”

高频动作最好沉淀为：

- `skill`
- `脚本`
- `Markdown 文档`
- `结构化 csv/json`

这一步对长期效率提升非常大。

### 4. 允许它并行和自迭代

你在很多 prompt 里都会明确写：

- 这是复杂任务，可以用 `subagent`
- 可以并发跑
- 可以修改参数后重新编译、重新运行
- 可以先做 side task

这相当于显式授权 Codex 从“被动答题”切换成“主动推进任务”。

### 5. 及时加边界条件

这类边界条件非常有用：

- “只读，不改文件”
- “trace 已经 2GB 了，别再继续打这个日志”
- “先把中间结论写到文档里”
- “skill 不要删，文档不要删”

它能显著减少任务发散。

## 七、给组里同学的几点建议

如果只想记住一句话，我会建议记住下面这句：

> 不要把 Codex 当成“回答问题的聊天机器人”，而要把它当成“能推进研究流程的执行代理”。

具体落地时，可以优先做到下面几件事：

1. 重复做两次以上的流程，就考虑沉淀成 skill。
2. 复杂任务不要只问“一步”，要把“分析-验证-再分析”的闭环一起交给 Codex。
3. 设计类问题先收敛文档，再进实现，不要一开始就直接写代码。
4. 能给路径就给路径，能给现成文档、checkpoint、CI 归档就不要只给一句口头描述。
5. 让 Codex 顺手写 Markdown、CSV、JSON，这些中间产物后面都会变成复盘材料。
6. 对耗时实验，优先采用“本地先做小验证，远端 CI 跑大实验，回来再统一分析”的方式。

如果把这几条做好，Codex 的角色就会从“帮忙答题”升级成“帮忙推进研究”。

## 八、一个推荐的协作模板

如果组里同学想直接照着用，我建议把复杂任务的 prompt 尽量写成下面这种结构：

```text
这是个复杂任务，你可以使用已有 skills 和 subagent。

目标：
1. 先读取已有文档/结果，确认当前问题定义。
2. 再分析本地或 CI 数据，定位最关键的证据。
3. 如有必要，修改参数或代码，重新编译并跑一个最小可复现实验。
4. 把中间结论持续记录到一个 Markdown 文档中。
5. 如果发现这套流程后续会重复使用，请顺手沉淀成一个 skill 或脚本。

输入路径：
- 文档：
- 配置：
- checkpoint / binary：
- CI run / archive path：
- 本地分析脚本：

约束：
- 复杂任务可并行。
- 只在必要时打大 trace。
- 优先给我可复用产物，而不是一次性结论。
```

这个模板的关键，不在于字面完全照抄，而在于它把几个最重要的东西都说清楚了：

- 目标是什么
- 输入材料在哪里
- 是否允许自迭代
- 是否允许并行
- 希望最终留下哪些可复用产物

只要这几个信息明确，Codex 的执行质量通常会比“帮我看看这个”高很多。

---

## 九、对这批 prompts 的一个总判断

回头看这批记录，我觉得最有价值的地方不是某一条 prompt 写得多漂亮，而是逐渐形成了一个稳定协作模式：

- 用 skills 固化套路
- 用文档固定中间结论
- 用 Codex 跑执行闭环
- 用 CI 做异步实验
- 用本地脚本做结果解释

这套方法一旦稳定下来，Codex 就不只是“帮忙写代码”，而是真的能参与推进研究。
